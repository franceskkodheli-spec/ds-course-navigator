Remote Procedure Call (RPC) and Remote Method Invocation (RMI) are two fundamental paradigms for enabling communication in distributed systems. Both aim to make a remote interaction look like1 a local call, thus hiding network complexity from the programmer. They rely on middleware that manages serialization, message transmission, and the transparent invocation of remote operations.

RPC: Communication Model and Operation

RPC follows the procedural paradigm: a client invokes a remote procedure on a server. Its main features are:

  • Synchronous Interaction: By default, RPC is blocking3: the client waits until the server completes the procedure and returns a result.
  • Point-to-Point Model: Communication is between one client and one server.
  • Operation Steps: The process includes calling the client stub, marshalling parameters, sending them via the transport layer (TCP/UDP), unmarshalling on the server, invoking the procedure, and returning the result along the same path.
  • Optimizations and Variants: Variants such as asynchronous RPC, batched RPC, or lightweight RPC (LRPC) have been introduced to address performance and flexibility issues.

RMI: Communication Model and Operation

Remote Method Invocation (RMI) extends the RPC paradigm into the object-oriented domain. Its main features are:

  • Object-Oriented Interaction: Instead of calling procedures, clients invoke methods on remote objects, preserving the semantics of object-oriented programming (encapsulation of state and behavior).
  • Stub-Skeleton Architecture:
    • Proxy (Stub): On the client side, method invocations are intercepted by a stub (proxy), which marshals the call parameters into network messages.
    • Skeleton: On the server side, the skeleton unmarshals incoming messages, reconstructs the method invocation, and dispatches it to the actual remote object implementation.
  • RMI Registry: A naming service that maintains the mapping between remote object identifiers (URIs) and the corresponding remote object references, enabling clients to locate and bind to objects dynamically.
  • Reference Semantics: RMI abstracts distributed complexity by allowing remote objects to be manipulated as if they were local. This includes maintaining aliasing relationships between references, so that multiple references pointing to the same remote object preserve their identity semantics.

The Problem of Parameter Passing in RPC and RMI

One of the main challenges in distributed systems is how to pass parameters across processes and machines. While local programming languages support both pass-by-value and pass-by-reference, in distributed contexts each approach has important limitations.

RPC and the problem of pass-by-reference

  • Pointers are meaningless across address spaces: a memory address valid on the client2 has no meaning on the server.
  • Copy/restore (value-result) can work for arrays, but not for complex data structures such as lists or trees.
  • Remote dereferencing would require every pointer access to become a network call, leading to unacceptable overhead.
  • For these reasons, implementations like Sun RPC only allow4 pass-by-value, explicitly forbidding pointers.

RMI and the solution for references

  • RMI introduces remote objects, identified by a unique remote reference (ObjID).
  • On the client, a stub acts as a proxy that marshals method calls into messages; on the server, a skeleton unmarshals them and invokes the real object.
  • The RMI registry maintains the mapping between URIs and remote objects.
  • This model preserves aliasing semantics: multiple references to the same remote object still point to the same entity, and updates are visible across clients.

Limitations of pass-by-value in RMI

  • Serialization overhead: complex objects may be large and costly to copy.
  • Loss of sharing: two references to the same object locally become two independent copies remotely.
  • Consistency issues: the copied objects can diverge, leading to stale data and semantic mismatches.

Different Approaches in Technologies

  • Sun RPC: Simplifies the problem by allowing only pass-by-value. It uses XDR (External Data Representation) for platform-independent marshalling. Limitations include one input and one output parameter and no support for references.
  • Java RMI: Introduces two mechanisms:
    • Serializable objects are passed by value.
    • Remote objects are passed by reference, using proxies that redirect calls back to the original object on the client. This preserves reference semantics in distributed environments.
  • CORBA: Designed for multi-language and multi-platform systems. Supports both pass-by-reference and pass-by-value (with valuetypes). However, in heterogeneous contexts, passing by value is complex, as it would require transferring not only the object's state but also its code. The responsibility for semantic consistency lies on the programmer.

Issues and Limitations

Despite their usefulness, RPC and RMI have several issues:

  • Failure Semantics: Distinguishing between a crashed server, a slow server, or a network failure is difficult.
  • Performance: Marshalling, unmarshalling, and network transmission make remote calls inherently slower than local ones.
  • Transparency Pitfalls: While transparency simplifies programming, it may hide important distributed concerns (e.g., partial failures, latency).
  • Parameter Semantics: As discussed, reference passing is problematic in RPC, while value passing can be misleading in RMI.

Lightweight Remote Procedure Call (LRPC)

Lightweight RPC (LRPC) was introduced to reduce the overhead of standard RPC when client and server processes reside on the same machine. Using network protocols (e.g., TCP/UDP) in this case is inefficient because it requires multiple copies of data and expensive context switches. Main ideas are:

  • Shared memory buffers: Instead of sending data through the network stack, LRPC uses a shared memory region to directly pass parameters and results between client and server.
  • Optimized control transfer: The kernel performs a direct context switch between the two processes, avoiding unnecessary message passing through sockets.
  • Reduced copies: Standard RPC requires up to 4 data copies (client → kernel, kernel → network, network → kernel, kernel → server). LRPC reduces this to a single copy using shared buffers.

Message-oriented communication provides asynchronous alternatives to RPC and RMI, based on the exchange of unidirectional messages1 rather than synchronous request-reply. These models support asynchronous, persistent, and multipoint communication, and are fundamental in distributed systems.

Message Passing (MPI)

Message passing is the most direct form of message-oriented communication, often mapped onto the underlying network facilities of the operating system (e.g., sockets). The Message Passing Interface (MPI) is the de facto middleware standard, providing higher-level primitives for transient, asynchronous communication in parallel and distributed systems.

Main Characteristics

  • Communication Model: Transient communication: both sender and receiver must be active at the same time for message delivery. Communication occurs within a predefined group of processes.
  • Addressing: Explicit addressing using process identifiers. Each process in a group has a local ID, and the pair (groupID, processID) uniquely identifies source and destination.
  • Synchronicity: Provides both blocking and non-blocking primitives (e.g., MPI_send, MPI_recv, MPI_isend, MPI_irecv), enabling synchronous and asynchronous communication.
  • Multipoint Support: In addition to point-to-point, MPI supports collective operations such as MPI_Bcast (broadcast), MPI_Reduce, MPI_Scatter, and MPI_Gather.
  • Reliability: Fault tolerance is not supported natively; process crashes are considered fatal.

MPI is widely used in high-performance computing (HPC) environments, such as clusters and supercomputers, where efficient low-level but standardized communication between processes is required.

Message Queuing

Message queuing provides persistent, asynchronous point-to-point communication2. Messages are stored in queues until the receiver is ready to consume them, thus decoupling senders and receivers both in space and in time (similar to email).

Main Characteristics

  • Communication Model: Point-to-point, persistent, asynchronous. A message is guaranteed to be stored in the destination queue, though the middleware cannot guarantee how the consumer will handle it.
  • Architecture: Peer-to-peer with input/output queues managed by queue managers. Queue managers may act as relays (application-level routers) and are often organized in overlay networks for scalability and fault tolerance.
  • Addressing: Explicit but symbolic. Queues are identified by symbolic names, and a lookup service maps these names to network addresses.
  • Synchronicity: Asynchronous, enabling senders and receivers to operate independently.

Use Cases

  • Client-server architectures where clients need not stay connected for responses.
  • Task processing and workflow systems, where messages are distributed to workers asynchronously.
  • Load balancing by sharing a queue among multiple consumers.
  • Enterprise integration via message brokers, which convert message formats and integrate heterogeneous subsystems.

Publish-Subscribe

The publish-subscribe model provides asynchronous, multipoint, transient communication based on event notifications. Publishers emit events without knowing the subscribers, who declare their interest through subscriptions. An event dispatcher (centralized or distributed) delivers each event to all matching subscribers.

Main Characteristics

  • Communication Model: Multipoint, implicit, asynchronous, transient.
  • Addressing: Implicit and anonymous. Publishers are unaware of subscriber identities; routing is based on event content or topic.
  • Subscription Languages:
    • Topic-based: Subscribers register to predefined topics (similar to multicast groups).
    • Content-based: Subscriptions are expressed as filters or queries over event attributes, offering higher expressiveness but higher complexity.
  • Dispatcher Architecture:
    • Centralized dispatcher: simple but a single point of failure.
    • Distributed brokers in overlay networks: acyclic graphs (subscription or message forwarding), hierarchical trees, or DHT-based routing for scalability.
  • Decoupling: Provides strong spatial, temporal, and flow decoupling, making it well-suited for dynamic environments.

Use Cases

  • Event notification and real-time alerts.
  • Monitoring and logging in distributed systems.
  • IoT data streaming and social media feeds.
  • Coordination of microservices in event-driven architectures.

Why Middleware Should Support Both Models

A modern Message-Oriented Middleware (MOM) often supports both queuing and publish-subscribe, because they are complementary3:

  • Different application needs: queuing is suited for reliable task distribution and request-response workflows, while publish-subscribe fits real-time notifications, monitoring, and scalable event dissemination.
  • Hybrid scenarios: events may be broadcast via publish-subscribe for immediate notification, while also enqueued for guaranteed and reliable processing by worker services.
  • Shared infrastructure: both models rely on common middleware components such as message routing, serialization, connection management, security, monitoring, and fault tolerance.

Supporting both models allows a MOM to cover a broad spectrum of distributed applications, offering developers the flexibility to choose the most appropriate style (or even combine them) depending on the interaction pattern required.

The publish-subscribe (Pub/Sub) model is an event-based communication paradigm where components interact indirectly by producing and consuming event notifications1. Publishers generate and send events, while subscribers express their interest in specific classes of events. Messages are routed by an intermediary component called the event dispatcher, which forwards each event to all subscribers whose subscriptions match its attributes. Communication in Pub/Sub systems is:

  • Message-based: Interaction is entirely based on event messages.
  • Asynchronous: Publishers do not block waiting for acknowledgments.
  • Multicast (Multipoint): Each published event can reach multiple interested subscribers.
  • Implicit Addressing and Anonymity: Publishers are unaware of the identity or number of subscribers; routing is handled by an event dispatcher.
  • Transient: Both publisher and subscriber must be active simultaneously for event delivery.

This model provides a high level of decoupling in space, time, and synchronization, making it particularly suitable for dynamic, scalable environments such as microservices and IoT systems.

Distributed Dispatcher Architectures

For scalability and fault tolerance, event dispatchers are often implemented as a network of brokers forming an overlay structure, typically a tree or a Directed Acyclic Graph (DAG). This means that there are no routing loops between brokers, ensuring that each event follows a single, well-defined path from publishers to subscribers.

Approaches in acyclic graphs

  • Message Forwarding: Each broker maintains routing or subscription tables that record which neighbors or clients are interested in specific topics. When an event arrives, the broker forwards it only to those neighbors with matching subscriptions, not to every connected node. This minimizes unnecessary traffic and delivers messages only to interested parties.
  • Subscription Forwarding: When a client subscribes to a topic, the broker sends this subscription to its neighboring brokers in the acyclic overlay. Each broker records which of its neighbors are interested in that topic, creating paths that will later be used to deliver events. In this way, subscriptions spread through the network and build the routes for message forwarding. Each subscription is propagated only once per link, avoiding loops and redundant traffic, while optimizations such as fusion and summarization reduce overhead by merging similar subscriptions.
  • Hierarchical Forwarding: Brokers are organized as a hierarchy, usually forming a tree. Clients connect to brokers at different levels, and brokers connect upwards to a single root broker. When a client subscribes to a topic, the subscription moves upward through the tree until it reaches the root. Each broker keeps track of which of its child brokers or clients are interested in that topic. When an event is published, it travels downward from the root, following only the branches that have active subscriptions. This ensures that messages are delivered only where they are needed, reducing network traffic and improving scalability.

DHT-Based Distributed Dispatching

A Distributed Hash Table (DHT) is a decentralized data structure that allows efficient and scalable lookup and storage of key-value pairs across a set of distributed nodes. Each node is responsible for a portion of the global keyspace and can quickly locate the node responsible for any given key, usually in O(log N) hops, where N is the number of nodes.

DHTs form an overlay network, often organized as a logical ring. Each node is assigned a unique identifier (usually a hash of its IP address), and each data item is identified by a key (a hash of its content or topic). The node responsible for a key is typically the one whose ID is the smallest identifier greater than or equal to that key (called the successor).

To store or retrieve data, the system computes the hash of the key and routes the request toward the responsible node using routing tables (e.g., finger tables in Chord). This routing mechanism ensures efficient, logarithmic-time lookups across the network.

Nodes can join or leave at any time. When this happens, routing tables and key assignments are automatically updated so that all keys remain available and correctly routed, maintaining scalability and fault tolerance.

Operation

  • Subscription:
    • 1. Compute hash H_S of the topic S.
    • 2. Route subscription to the node responsible for key H_S (the successor node).
    • 3. Along the path, nodes store backpointers to support later event forwarding.
  • Publication:
    • 1. Compute hash H_S of the same topic.
    • 2. Route the publication to succ(H_S).
    • 3. The event is sent back through the stored backpointers to all matching subscribers.

Limitations

While Distributed Hash Tables (DHTs) are highly efficient for exact-match lookups, they have several important limitations:

  • Limited query expressiveness: DHTs cannot efficiently support range queries or complex predicate filtering. They only handle lookups of the form "find value with key k".
  • Poor support for content-based routing: DHTs work well for topic-based publish/subscribe systems (exact matching), but not for content-based models where subscriptions depend on message attributes or conditions.
  • No awareness of network locality: Routing decisions ignore physical proximity, so messages may travel through long or inefficient network paths.
  • Sensitivity to churn: Since DHTs rely on structured routing tables, frequent node joins or failures can disrupt routing consistency and increase maintenance overhead.

The stream-oriented communication model is fundamental in distributed systems where data is transmitted as a continuous sequence (a stream), such as in video conferencing, audio streaming, or real-time sensor data feeds. In this model data is transmitted as a continuous sequence of data units (bytes, samples, or frames), often subject to temporal constraints. Actually, time usually does not impact the correctness of the communication1, just its performance. We can have different types of transmission:

  • Asynchronous: Data units are sent sequentially, maintaining order but without timing guarantees.
  • Synchronous: Each unit must be delivered within a maximum end-to-end delay.
  • Isochronous: Delivery is constrained by both minimum and maximum delay bounds, controlling jitter (delay variation).

Quality of Service (QoS) in Streams

In this model, ensuring Quality of Service (QoS) is crucial, as requirements go beyond simple delivery: they can involve constraints on latency, jitter, bandwidth, and synchronization. QoS represents the system's ability to meet predefined performance levels during transmission. Key parameters include:

  • Required bit rate (minimum bandwidth).
  • Maximum setup time (stream startup delay).
  • Maximum end-to-end delay (latency).
  • Maximum jitter (delay variation).
  • Stream synchronization (alignment between correlated streams, e.g., audio and video).

Approaches to Address QoS Issues

Since most network protocols (e.g., IP/TCP) provide only best-effort service, QoS must be ensured through multiple layers: network-level and application/middleware-level strategies.

1. Network Layer Approaches

  • DiffServ (Differentiated Services): Marks packets with priority bits in the IP header to request preferential treatment from routers. Effectiveness depends on network support and is often ignored in public Internet.
  • Bandwidth Reservation: Protocols (like RSVP) can try to reserve resources along the network path. Rarely usable on the open Internet.

2. Application and Middleware Strategies

These are the most practical solutions in real distributed systems, where end-to-end guarantees are limited.

  • Buffering: Incoming data is temporarily stored before playback, absorbing delay variations (jitter) at the cost of initial latency.
  • Forward Error Correction (FEC): Adds redundant packets to allow reconstruction of lost or corrupted data without retransmission. Example: sending 10 data packets + 2 redundant packets allows recovery from up to 2 losses.
  • Interleaving: Shuffles data units from different parts of the stream into each packet. If a packet is lost, only small pieces from multiple parts of the stream are missing (i.e. in audio, this results in much less noticeable glitches).
  • Adaptive Streaming: The sender dynamically adjusts bitrate or resolution according to current network conditions, maintaining continuity at the expense of quality.
  • Synchronization Mechanisms: Essential when multiple correlated streams (e.g., audio and video) must remain aligned; implemented at sender or receiver level.

A naming system is a fundamental component of distributed systems, defining how entities are uniquely identified, located, and accessed. Entities may represent hosts, users, files, or services, and are typically accessed through one or more access points, each characterized by an address. While an address is a particular kind of name, it is often undesirable to use addresses directly as names—especially in the presence of mobility—since access points may change during an entity's lifetime. Instead, location-independent names are preferred, allowing the entity to be referred to independently of its current position.

The process of obtaining the address of a valid1 access point given a name is called name resolution. The strategy used for name resolution depends on the nature of the adopted naming scheme, which can generally be categorized as flat, structured, or attribute-based naming.

Flat naming

In a flat naming system, names are simple, structureless identifiers—typically opaque strings that have no internal meaning or hierarchy. Because names do not encode any structural information, resolution must rely on auxiliary mechanisms.

Flat naming mechanisms. Several techniques can be used to locate entities given their flat names:

  • Broadcast or multicast search: a client sends a "find" request containing the sought name as a broadcast (to every node) or as a multicast (to a subset of nodes belonging to a group); each node receives and processes the request, but only the node holding the matching name responds to the client with the address or location of the entity. This works well on small-scale networks but scales poorly due to excessive traffic.
  • Home-based approaches: each entity is associated with a stable home node that stores its current address. The home is queried for location information, but this adds latency and lacks scalability.
  • Distributed Hash Tables (DHTs): entities are stored in a structured overlay network, where each node is responsible for a portion of the name space. A hash function maps each name (or key) to an identifier, which determines the node responsible for it (e.g., in Chord, identifiers are arranged in a ring). When resolving a name, the query is routed through intermediate nodes toward the responsible one, typically requiring only O(log N) hops, providing scalability in large, decentralized systems.
  • Hierarchical flat naming: the overall system is partitioned into domains (like departments, subnets, or geographical regions). Each domain is responsible for knowing about entities within it, mapping flat names to locations or addresses. Lookup requests start locally and move up the root, then down into subdomains.

Hierarchical flat naming

The network is logically divided into domains, each associated with a directory node responsible for managing the entities it contains.

  • The root domain spans the entire distributed system and must store references to all entities or to the subdomains containing them.
  • Each leaf domain (typically a LAN or local subsystem) holds the actual address of the entities it manages.
  • Internal directory nodes maintain pointers to entities within their subtree, allowing requests to be recursively forwarded to the appropriate subdomain.

Lookup. The lookup process typically begins where the client resides. If the entity is not found locally, the request moves3 upward to the parent domain, which may either resolve the request or forward it downward into the proper subdomain. This two-way traversal—up to the root, then down to the leaf—leverages locality but increases resolution latency in large systems.

Operations and mobility. When entities move or change location, updates must propagate through several domains. Insertions can be performed bottom-up, ensuring fast availability, while deletions proceed upward until other references or replicas are found.

Scalability limitations. Although this hierarchical organization mitigates the inefficiency of pure flat naming, it still faces scalability constraints. The root node remains a critical bottleneck, since it must contain information about all entities or domain mappings. Basically, the more we go up, the bigger is the database we need to know where the target node is. Moreover, caching is generally inefficient.

Structured naming

A structured naming system organizes names into a well-defined name space—a labeled graph composed of:

  • Leaf nodes, representing named entities, which store information such as identifiers or addresses.
  • Directory nodes, which contain labeled outgoing edges to other nodes, thus defining a hierarchical structure.

Entities are referred to through path names, such as /alpha/beta/gamma, which can be absolute (starting from the root) or relative to another node. Multiple path names may refer to the same entity via hard links, or a leaf may store a symbolic link to another name.

Hierarchical distribution. The namespace is partitioned into multiple layers:

  • Global level: top-level domains shared across administrative authorities.
  • Administratival level: mid-level domains managed independently by organizations.
  • Managerial level: local domains within single organizations, typically containing dynamic or frequently changing data.

Each zone is served by a dedicated name server, responsible only for its portion of the hierarchy. Higher-level servers know only which subordinate servers manage the next part of the path, enabling true distribution of responsibility.

Name resolution. In structured naming, resolution follows the structure of the name: the client starts from the root and traverses the hierarchy according to the components of the path (e.g., .it → polimi.it → ftp.polimi.it). This allows the system to delegate each part of the lookup to the responsible authority, providing scalability and locality.

Name resolution can be performed in two main ways: (i) Recursive resolution, where each contacted server takes full responsibility for forwarding the query to the next server until the final answer is found (the response then propagates back along the same chain); and (ii) Iterative resolution, where the client itself iteratively contacts each referred server based on the hints returned by the previous one. Recursive resolution reduces the number of client interactions but increases the load on servers, while iterative resolution is more scalable and is typically used in systems such as DNS between resolvers and higher-level servers.

Performance and scalability benefits. Structured naming achieves scalability and efficiency through several design choices:

  • Guided resolution: the path expressed in the name2 directly determines the sequence of servers to contact, avoiding redundant upward traversals.
  • Load distribution: each server manages a well-defined zone, removing the single-root bottleneck typical4 of hierarchical flat naming.
  • Localized updates: changes within a local domain affect only the servers in that domain, while upper levels remain stable.
  • Caching and replication: structured systems exploit aggressive caching (with TTL-based validity) and replicated root servers—often accessed via IP anycast—to improve response time and resilience.

When structure is not needed. Despite its advantages, the structured approach introduces higher implementation complexity. If lookups are almost exclusively local—resolved within a single administrative domain—a flat hierarchical organization may perform equally or even more efficiently, since it reaches the responsible node directly without traversing global name servers, while in a structured system, even with caching, queries may need to ascend toward the root once cache entries expire (according to their TTL). Structured naming becomes advantageous only when resolution must scale across multiple domains or administrative boundaries.

Attribute-based naming

In attribute-based naming, entities are not identified by unique names but5 by a set of attributes that describe their properties. Queries specify desired attribute values, and the naming service returns all entities matching those criteria. Such systems are typically implemented as directory services using database technologies.

Example: LDAP. The Lightweight Directory Access Protocol (LDAP) combines hierarchical and attribute-based naming6. Each directory entry consists of a collection of <attribute, value> pairs and is identified by a Relative Distinguished Name (RDN). Entries are organized into a Directory Information Tree (DIT), which is hierarchically partitioned across multiple Directory Service Agents (DSAs). For example:

/C=NL/O=Vrije Universiteit/OU=Comp.Sc./CN=Main Server

LDAP extends the structured naming approach by adding powerful search capabilities, allowing complex queries such as:

search("(C=NL)(O=Vrije Universiteit)(OU=*)(CN=Main Server)").

This means:

  • C=NL restricts the search to entries located in the country Netherlands;
  • O=Vrije Universiteit selects entries belonging to the organization Vrije Universiteit;
  • OU=* indicates that any organizational unit (OU) is acceptable, as denoted by the "*";
  • CN=Main Server looks for an entry whose common name (CN) is Main Server.

The result of the search is therefore the directory entry (or entries) describing the main server of the Vrije Universiteit in the Netherlands, regardless of the specific subunit in which it is located.

Scalability considerations. While attribute-based naming provides high flexibility and expressiveness, it scales less efficiently than structured naming due to several intrinsic limitations:

  • Query routing complexity: attribute-based queries do not follow a fixed resolution path, as in hierarchical naming. A single search may involve multiple Directory Service Agents (DSAs) and thus generate significant network and server load, while structured naming performs predictable, tree-based traversals.
  • Lack of locality: in structured systems such as DNS, lookups are typically confined within a specific domain or subtree, enabling efficient caching and localized resolution. In contrast, the same attribute can appear in multiple domains, forcing LDAP to perform distributed searches across several servers and reducing the effectiveness of caching.
  • Server load and replication: structured systems assign each namespace segment to a small number of authoritative servers, allowing extensive use of caching, replication, and load balancing. Attribute-based systems often require queries or updates to be processed by many servers simultaneously, increasing coordination complexity and introducing bottlenecks as the directory grows.
  • Scale of directory entries: attribute-based directories may contain millions of entries and support broad queries (e.g., "find all users named Mario in Italy"). Such queries are inherently less selective than path-based lookups that resolve a single, well-defined node, leading to higher computational and communication costs.

As a result, although attribute-based naming offers powerful search capabilities, its distributed search nature and lack of strict hierarchy limit its scalability compared to structured, path-oriented naming systems.

Removing unreferenced entities

The automatic removal of entities that are no longer referenced, known as distributed garbage collection (DGC), is essential to free resources in long-running distributed systems1. Unlike centralized garbage collection, DGC is challenging due to the lack of global knowledge about which process still holds a reference and the possible presence of network failures or partitions. This problem is particularly relevant in platforms such as Java RMI, where remote objects must be destroyed once no client references remain.

The three main approaches to DGC are: Reference Counting, Reference Listing, and Distributed Mark-and-Sweep. Each method differs in its scalability, reliability, and ability to handle faults or circular references.

Reference Counting. In the reference counting method, each distributed object stores a counter representing how many entities currently reference it. Whenever a process creates a new reference, it sends an increment message; when a reference is deleted, a decrement message is sent. If the counter reaches zero, the object can be safely removed. However, this method is fragile in distributed contexts:

  • Race conditions: when a reference is passed between processes, message reordering can cause premature deletion—if a "decrement" arrives before the corresponding "increment".
  • Communication cost: preventing races requires at least three messages per update (increment, acknowledgment, and confirmation), severely limiting scalability.

Weighted reference counting mitigates this issue by dividing a total reference weight among processes, ensuring that weights rather than counts are adjusted. While this avoids acknowledgment races, it constrains the total number of references and introduces complexity in maintaining weight balance.

Reference Listing. The reference listing approach replaces counters with explicit tracking of which clients hold references. Each remote object maintains a list of active proxies, identified by their process IDs or network endpoints. Operations for adding or removing entries are idempotent, meaning they can be safely retried after failures without inconsistencies.

In distributed environments, the list's consistency is maintained by periodic pings or lease renewals. If a client fails to renew its lease before expiration, its entry is removed and the reference is considered invalid. This method is used in systems such as Java RMI, where a client creates a proxy only after receiving an acknowledgment from the server. RMI employs renewable leases to avoid permanently lost references and to clean up inactive ones efficiently.

Although more fault-tolerant than reference counting, reference listing still faces scalability issues: maintaining long lists and periodic checks creates overhead in systems with a large number of clients or high churn.

Distributed Mark-and-Sweep. Mark-and-sweep is a tracing-based algorithm that identifies unreferenced objects by exploring the reachability graph of entities. It proceeds in two phases:

  • 1. Mark phase: all entities start "white." Objects reachable from the root set are successively marked "gray," and then "black" once all their outgoing references have been examined.
  • 2. Sweep phase: once no gray nodes remain, all white (unmarked) entities are unreachable and can be safely deleted.

In distributed systems, this approach requires that the global graph of references remain stable during the marking phase. To guarantee this, synchronization or distributed transactions are often needed, which can temporarily block updates or communications. Therefore, while this technique can reclaim cyclic dependencies—something counting schemes cannot—it is rarely practical for large or dynamic distributed systems due to its coordination cost.

Comparison and trade-offs. Each approach represents a trade-off between simplicity, fault tolerance, and scalability:

  • Reference counting is simple and fast for small-scale, reliable systems but scales poorly and cannot handle cycles.
  • Reference listing improves reliability and consistency under failures, and is suitable for moderate-scale systems such as RMI. Also, not connected nodes are a problem.
  • Distributed mark-and-sweep provides the most complete cleanup but incurs high overhead and requires strong synchronization, making it suitable only for tightly coupled or smaller distributed environments.

In practice, hybrid solutions combine reference listing for ordinary cases with occasional global tracing to handle cycles, balancing efficiency and completeness in distributed garbage collection.

In a distributed system, each node maintains its own local clock, which may drift over time due to hardware imperfections. Without synchronization, these drifts cause inconsistencies that can affect event ordering, coordination, and correctness of distributed applications. Clock synchronization can be approached in two complementary ways: by aligning physical clocks to a real reference time (e.g., UTC), or by using logical clocks that provide only a consistent ordering of events.

Goals and Challenges

The goal of clock synchronization is to ensure that all processes in the system share a common notion of time1, either real or logical. The main challenges are:

  • Clock drift: hardware timers are imperfect and diverge over time at a rate bounded by a constant ρ.
  • Message delays: communication latency varies and cannot be exactly predicted.
  • Absence of a global reference: there is no shared physical clock in distributed systems.

Depending on the application, synchronization aims for either:

  • Accuracy: alignment with an external reference (e.g., UTC).
  • Agreement: internal coherence among all nodes (relative synchronization).

Main Techniques for Physical Clock Synchronization

Several algorithms have been developed to handle clock synchronization at different scales and accuracy requirements2.

Cristian's Algorithm (1989)

A simple client–server protocol in which each client periodically asks a trusted time server for the current time. The steps are:

  • The client records its local time T0 and sends a request to the server.
  • The server replies with its current time Ts.
  • Upon receiving the reply, the client records its local time T1.
  • The client estimates the round-trip delay as RTT = T1 − T0.
  • The client sets its local clock to Ts + RTT/2.

To reduce error, the process is repeated several times, using the minimum RTT measured over several attempts3. To avoid issues like clock going backward, adjustments are applied gradually (e.g., slightly increasing/decreasing the clock rate until synchronized). Cristian's algorithm assumes that message delays are symmetric and relatively small compared to the required precision. If this assumption fails, the error grows.

Use in Wireless Sensor Networks (WSNs)

Cristian's algorithm is generally unsuitable for Wireless Sensor Networks, since these systems are composed of battery-powered and resource-constrained nodes connected through unreliable wireless links. Communication delays are often unpredictable due to interference and congestion, making round-trip time estimations inaccurate. Moreover, WSNs usually lack a permanently reachable, high-precision time server, which is a core assumption of Cristian's approach.

Nevertheless, the algorithm can be employed in very specific scenarios, provided that the following conditions hold:

  • A stable and accurate gateway (sink node) is available to act as a reference time server for the network.
  • The network topology ensures low-latency and relatively symmetric communication paths between the nodes and the gateway.
  • The required synchronization accuracy tolerates the variability of wireless transmission delays.
  • Energy consumption is acceptable, or synchronization is executed infrequently to reduce overhead.

Berkeley Algorithm (1989)

A refinement of Cristian's algorithm, designed for systems without an authoritative external clock. The server polls all nodes, collects their local times, computes an average (excluding outliers), and instructs each node to adjust its clock accordingly. This ensures that all nodes agree on a common time, although it might not be accurate with respect to UTC. Adjustments are performed gradually to prevent the clock from jumping backward.

Network Time Protocol (NTP)

NTP is the standard Internet protocol for clock synchronization. It organizes nodes into a hierarchy of strata:

  • Stratum 0: atomic or GPS reference clocks.
  • Stratum 1: servers directly connected to reference clocks.
  • Stratum 2+: servers that synchronize with higher strata.

Each node estimates offset and delay using multiple timestamp exchanges and statistical filtering, achieving millisecond-level accuracy over the Internet and microsecond-level on local networks. NTP is scalable and fault-tolerant thanks to its redundant, hierarchical structure, but it is more complex than simpler algorithms.

GPS-based Synchronization

Nodes equipped with GPS receivers can directly obtain precise UTC time by triangulating signals from satellites. This provides nanosecond accuracy but requires additional hardware and clear satellite visibility, making it suitable for servers, base stations, and data centers.

Describe scalar (Lamport) clocks and how they implement a totally ordered multicast

Scalar clocks, also known as Lamport clocks, are a logical time mechanism used to order events in a distributed system where no global clock exists. Instead of measuring physical time, they provide a numerical representation of event order that respects the happens-before relation (→). They are a fundamental tool to guarantee consistent event ordering1 and are often used to implement communication primitives such as totally ordered multicast.

How Scalar Clocks Work

Each process pi maintains a local integer clock Li, initialized to zero. The update rules are as follows:

  • For each local event at pi: Li := Li + 1
  • When sending a message m: Li := Li + 1; attach the timestamp Tm = Li
  • When receiving a message with timestamp Tm: Li := max(Li, Tm) + 1

This guarantees that for any two events e and e′:

e → e′ ⇒ L(e) < L(e′)

Lamport clocks thus preserve causal ordering, although they do not detect concurrency.

To transform this partial order into a total order, timestamps are paired with process identifiers, forming tuples (L, PID). These pairs can be compared lexicographically, ensuring a unique total order of all events in the system.

Totally Ordered Multicast Using Scalar Clocks

A totally ordered multicast ensures that all processes deliver all multicast messages2 in the same global order. This property is essential for maintaining consistency in replicated data and ensuring that all nodes observe the same sequence of updates.

Protocol Overview:

  • Sending: When a process wants to multicast a message, it increments its Lamport clock and attaches the tuple (Li, PIDi) to the message, then multicasts it to all processes (including itself).
  • Receiving and Queueing: Upon receiving a message, a process updates its local clock according to the Lamport rule and stores the message in a local queue ordered by the pair3 (L, PID).
  • Acknowledgment: Each process sends an acknowledgment to all other processes (or uses reliable FIFO communication). These acknowledgments are used to ensure that every process has received the same set of messages.
  • Delivery Condition: A message can be delivered to the application only when:
    • It is at the head of the queue (has the smallest timestamp and process ID).
    • All processes have received that message (i.e., all corresponding ACKs have been received).
  • Delivery: When these conditions are satisfied, the process delivers the message to the application and removes it from the queue.

Example:

Consider three processes P1, P2, P3 participating in a replicated banking application. Each process maintains a local copy of the account balance, and the goal is to ensure that all copies are updated in the same global order regardless of the order in which messages are sent or received.

Assume the following sequence of events:

  • P1 issues an operation "deposit 50" and multicasts it to all processes.
  • P2 issues an operation "add 1% interest" and multicasts it to all processes.

Initially, all Lamport clocks are set to zero:

L1 = L2 = L3 = 0

The protocol evolves as follows:

  • When P1 prepares to send its message, it increments its clock to L1 = 1. It then attaches the timestamp (1,1) (clock value 1, process ID 1) and multicasts the message deposit 50 to all.
  • Similarly, when P2 prepares its message, it increments its clock to L2 = 1 and multicasts the message add interest with timestamp (1,2).

As messages are received:

  • When P2 receives P1's message deposit 50 with timestamp (1,1), it updates its clock as L2 := max(L2,1) + 1 = 2 and stores the message in its local queue.
  • When P1 receives P2's message add interest with timestamp (1,2), it updates its clock as L1 := max(L1,1) + 1 = 2 and also stores the message in its queue.
  • P3 receives both messages (in any order) and performs the same clock update rule. After receiving both messages, it may have L3 = 2.

At this point, all processes have received both messages and have inserted them into their respective local queues, which are sorted lexicographically by timestamp and process ID:

ProcessLocal message queue (ordered by (L, PID))
P1[ (1,1): deposit 50 ], [ (1,2): add interest ]
P2[ (1,1): deposit 50 ], [ (1,2): add interest ]
P3[ (1,1): deposit 50 ], [ (1,2): add interest ]

Afterward, each process multicasts an acknowledgment (ACK) for every message it receives. In this example, the message deposit 50 (timestamp (1,1)) satisfies the conditions to be delivered first and is therefore delivered first by every process. After it is delivered and removed from the queue, the next message add interest (timestamp (1,2)) becomes eligible for delivery.

This example shows that, although multicasts may be received in different physical orders, Lamport clocks combined with process identifiers and acknowledgment-based delivery ensure that all processes deliver the same global sequence of updates, preserving strong consistency across replicas.

Assumptions for Correctness

The correctness of the scalar-clock-based totally ordered multicast relies on the following assumptions:

  • Reliable communication: All messages and acknowledgments are eventually delivered (no loss or duplication).
  • FIFO channels: Messages between any two processes are received in the order sent, simplifying delivery conditions.
  • No process crashes: All processes remain active until the multicast operation is complete.
  • Stable group membership: The set of participating processes is known and fixed during message exchange.

Violating these assumptions (e.g., due to message loss or process failure) may lead to inconsistent deliveries or indefinite blocking, as some processes may wait forever for missing acknowledgments.

Describe vector clocks and how they implement a causally ordered broadcast

Vector clocks extend Lamport's scalar clocks to provide both causal ordering and detection of concurrency between events. They are particularly useful to implement causally ordered broadcast, a communication primitive that guarantees that messages are delivered in an order consistent with the causal2 dependencies among events, but allows concurrent (independent) messages to be delivered in different orders at different processes.

Principle of Vector Clocks

Each process Pi maintains a vector clock Vi[1..N], where N is the number of processes in the system. Each entry Vi[j] represents the process Pi's knowledge of how many events have occurred at process Pj.

Clock update rules:

  • Initially, all vector clocks are set to zero: Vi[j] = 0 for all i, j.
  • On a local event at Pi: increment its own counter Vi[i] := Vi[i] + 1.
  • On sending a message m: increment Vi[i], attach a copy of the vector Vi to m.
  • On receiving a message with timestamp Vm: For each component k, set Vi[k] := max(Vi[k], Vm[k]), and then increment Vi[i] by one to record the receive event.

Two events e and e′ are causally related if and only if V(e) < V(e′), meaning V(e)[j] ≤ V(e′)[j] for all j and V(e) ≠ V(e′). If neither V(e) < V(e′) nor V(e′) < V(e), the events are concurrent.

Causally Ordered Broadcast Protocol

If two events are causally related, everybody must see the message in the same exact order. To achieve this, each process broadcasts and delivers messages according to the following rules:

On broadcast:

  • Increment the local clock: Vi[i] := Vi[i] + 1.
  • Attach a copy of Vi to the message m.
  • Multicast m to all processes (including itself).

On receive:

When process Pj receives a message m with timestamp Vm from sender Pk, it can deliver the message to the application only if:

  • Vj[k] = Vm[k] − 1 (the previous message from the same sender has already been delivered), and
  • For all other processes l ≠ k: Vj[l] ≥ Vm[l] (all messages that causally precede m have already been delivered).

If these conditions are not yet satisfied, the message is buffered. The process periodically checks its buffer to deliver messages whose dependencies are now fulfilled.

This guarantees that no message is delivered before any of its causal predecessors1.

Example:

Assume three processes P1, P2, P3 each maintaining a vector clock [V[1], V[2], V[3]].

  • P1 sends m1 with timestamp [1, 0, 0].
  • P2 receives m1, sets its vector clock to [1, 1, 0], then broadcasts m2 with timestamp [1, 2, 0].
  • P3 receives m2. It can deliver m2 only after it has delivered all messages that P2 had already seen when it sent m2—in this case, m1 from P1. Therefore, if P3 receives m2 before m1, it will buffer m2 until m1 arrives and is delivered.

This simple mechanism guarantees causal consistency: if one message influences another, all processes will deliver them in that causal order.

Assumptions for Correctness

The protocol requires the following system properties:

  • Reliable communication: Messages are neither lost nor duplicated.
  • FIFO channels: Messages between any pair of processes are received in the same order as sent.
  • No process crashes: All processes remain active until their messages are delivered.
  • Fixed group membership: The number of processes N is known and constant, since each vector has length N.

Relation to Totally Ordered Multicast

A totally ordered multicast, as implemented with Lamport scalar clocks, ensures that all processes deliver all messages in the same global order, even when messages are concurrent (i.e., causally unrelated). This implies causal order, since any total order that respects Lamport's "happened-before" relation automatically preserves causality. Therefore:

Total order ⟹ Causal order

However, the converse does not hold: causal order allows concurrent messages to be delivered in different orders at different processes.

Key Observation:

A totally ordered multicast can always replace a causally ordered one, since it imposes a stricter ordering that inherently satisfies causal constraints. However, doing so often introduces unnecessary synchronization and delays, since even concurrent operations are forced into a single global sequence. For many applications—such as collaborative editing or distributed discussion systems—causal ordering is sufficient and more efficient.

Replacing total order with causal order would therefore break consistency in applications that rely on a single, globally agreed message sequence—such as replicated databases, distributed transactions, or state machine replication. Causal ordering simply does not impose enough constraints to guarantee identical delivery order across all processes.

Mutual exclusion guarantees that, in a distributed system, at most one process at a time can access5 a shared resource or enter a critical section (CS). This prevents concurrent updates that could lead to inconsistent states. The challenge in distributed systems is that there is no global memory or clock — all coordination must rely on message passing3.

A correct distributed mutual exclusion algorithm must satisfy:

  • Safety: At most one process is in the critical section at any time.
  • Liveness: Every request is eventually granted (no deadlock or starvation).
  • Ordering: Requests are granted in the order they were made (respecting causal order).

Several techniques can be used to achieve this property, which differ in communication cost, fault tolerance, and implementation complexity1.

1. Centralized (Coordinator-Based) Algorithm

How it works: A single process acts as a coordinator:

  • A process sends a REQUEST to the coordinator when it wants to enter the CS.
  • If the resource is free, the coordinator replies with GRANT.
  • Otherwise, the coordinator queues the request.
  • Upon exit, the process sends a RELEASE message, and the coordinator grants the CS to the next waiting process.

Properties:

  • Message complexity: 3 messages per CS entry (request, grant, release).
  • Fairness: Requests served in FIFO order.
  • Bottleneck: Coordinator can become overloaded in large systems.

Assumptions:

  • The coordinator does not crash during operation.
  • Communication with the coordinator is reliable.
  • All processes know the identity of the coordinator.

If the coordinator fails, access to the resource halts until a new coordinator is elected or the system recovers.

Advantages:

  • Simple and efficient for small systems.
  • Low message overhead.

Disadvantages:

  • Single point of failure (the coordinator).
  • Limited scalability and fault tolerance.

2. Distributed Mutual Exclusion (Ricart–Agrawala Algorithm)

The Ricart–Agrawala algorithm eliminates the coordinator and relies on4 Lamport's scalar clocks to order requests fairly2 in a fully distributed way.

Protocol Overview:

  • Requesting access: When Pi wants to enter the CS:
    • It increments its clock: Li := Li + 1.
    • It multicasts a REQUEST(timestamp = Li, process ID = i) to all processes.
  • Receiving a request: When Pj receives REQUEST(T, i):
    • If Pj is neither requesting nor in the CS, it immediately replies REPLY.
    • If Pj is in the CS, it queues the request.
    • If Pj is also requesting, it compares timestamps: the smaller (T, ID) wins.
  • Entering CS: Pi enters the CS only after receiving a REPLY from every other process.
  • Releasing CS: Upon exit, Pi sends all pending REPLY messages to queued requests.

Example: If P1 and P3 both request simultaneously, their requests are timestamped (10,1) and (10,3). Since (10,1) < (10,3), all processes grant P1 first. P3 waits until P1 sends RELEASE and acknowledges its request.

Assumptions:

  • All processes and communication channels are reliable (no message loss or reordering).
  • Group membership is fixed and known to all.
  • No process crashes during participation.

Properties:

  • Safety: Only one process enters CS at a time.
  • Liveness: All requests are eventually granted.
  • Fairness: Requests served in Lamport timestamp order.

Advantages:

  • Fully distributed; no single point of failure.
  • Deterministic ordering and fairness.

Disadvantages:

  • High message overhead: 2(N − 1) messages per CS entry.
  • If any process or message fails, the algorithm may block indefinitely.

3. Token Ring Algorithm

How it works: Processes are organized in a logical ring:

  • A unique token circulates around the ring.
  • The token holder may enter the CS.
  • When done, the process passes the token to its successor in the ring.

Assumptions:

  • The logical ring structure is maintained and known to all processes.
  • The token is never lost or duplicated.
  • Processes in the ring do not fail or drop out unexpectedly.

If the token is lost or a process crashes, the system must detect and regenerate a new token.

Properties:

  • Message complexity: 1 message per CS entry (token passing).
  • Fairness: Each process gets the token in turn.
  • Delay: Worst-case N − 1 hops to reach the requesting process.

Advantages:

  • Very efficient when many processes request sequentially.
  • No central bottleneck.

Disadvantages:

  • Token loss or process crash requires detection and regeneration of a new token.
  • The order of token circulation does not necessarily match the real-time order of requests: a process Pi that issues a request later than another process Pj may still receive the token first, since token passing follows the logical ring order rather than request timestamps.
  • Fault detection and reconfiguration add implementation complexity.

Leader election is a fundamental distributed algorithmic problem whose goal is to ensure that all non-faulty processes in a distributed system agree on exactly one process to act as the1 leader (or coordinator), which is responsible for coordination tasks such as managing shared resources, synchronizing operations, or initiating global actions. Leader election is essential after a failure of the current coordinator or during system startup when no leader is known.

It generally relies on two fundamental assumptions:

  • Distinguishable nodes: Each process has a unique, totally ordered identifier (ID), ensuring a deterministic comparison between any two processes.
  • Closed group: The set of participating processes is known to all members — that is, every process knows the IDs of all others (no anonymous or dynamic membership during the election).

Main Leader Election Algorithms

The two classical approaches are the Bully algorithm and the Ring-based algorithm. Both rely on previous assumptions.

1. Bully Algorithm

We have some additional assumptions to solve the agreement problem:

  • Reliable links
  • The system is synchronous, so it is possible to understand who has crashed

Steps:

  • When a process Pi detects that the current leader is unresponsive, it initiates an election by sending ELECTION messages to all processes with higher IDs.
  • If no higher-ID process responds, Pi declares itself the leader and sends a COORD message to all lower-ID processes.
  • If a higher-ID process responds, that process takes over the election and repeats the procedure.
  • The highest-ID operational process ultimately wins and becomes the new leader (that's why it is called "Bully").
  • If a previously failed process with a higher ID recovers, it immediately reclaims leadership by starting a new election.

Advantages:

  • Simple and fast for small groups.
  • Always elects the highest-ID operational process (deterministic outcome).

Disadvantages:

  • High message overhead in large systems (message complexity is O(N²) in worst case).
  • Requires reliable crash detection (synchronous assumption).
  • May temporarily elect multiple leaders if timeouts are inconsistent.

2. Ring-Based Election Algorithm

As an additional assumption, processes are arranged in a logical ring (each knows its successor).

Steps:

  • A process that detects the leader's failure starts an election by sending an ELECTION message (containing its own ID) to its neighbor in the ring.
  • Each process appends its own ID to the message and forwards it to the next alive process.
  • When the message returns to the initiator, it determines the process with the highest ID and sends a COORDINATOR message around the ring to announce the new leader.

Advantages:

  • Message complexity is linear (O(N)).
  • Deterministic and easy to implement with ring topology.

Cut definitions

1Provide a formal definition of a 'cut' and a 'consistent cut'. Make an example of a consistent cut and a non-consistent one.

In a distributed system, the global state consists of the local state of each process and the messages currently in transit between them. Since there is no global clock or shared memory, it is impossible to capture this global state at a single physical instant. Instead, we define a cut as a conceptual snapshot that represents a possible combination of local process histories.

Definition of a Cut

Let a distributed system be composed of N processes P1, P2, ..., PN. Each process has a sequence of events ei0, ei1, ei2, ..., ordered by the local clock of Pi.

A cut C of the system is defined as the union of the histories of all its processes upto a certain event:

C = h1^k1 ∪ h2^k2 ∪ ··· ∪ hN^kN

where hi^ki = ⟨ei0, ei1, ..., ei^ki⟩ is the history of process Pi up to event ei^ki.

In other words, a cut represents a global "line" that intersects the timelines of all processes, selecting for each one all events that occurred before a certain point.

Figure: Two timeline diagrams of three processes P1, P2, P3 with time flowing left to right and messages m1, m2, m3 exchanged between them. In picture (a) a dashed line marks a consistent cut. In picture (b) a dashed line marks an inconsistent cut, where the cut includes the receipt of message m2 but not its send, so the sender of m2 cannot be identified with this cut.

Definition of a Consistent Cut

A cut C is said to be consistent if it respects the causal relationships between events1.

Formally:

∀e, f : (e ∈ C) ∧ (f → e) ⇒ f ∈ C

where f → e denotes the happens-before relation (Lamport, 1978).

This means that if an event e is included in the cut, then all events that causally precede it must also be included.

If a cut violates this property — for instance, by including the receipt of a message but not its send — it is said to be inconsistent. In Fig. 3.1 we can see an example where this happens ((b) picture).

The Chandy–Lamport Distributed Snapshot Algorithm

The Chandy–Lamport algorithm is a classic distributed protocol designed to record a consistent global state (or distributed snapshot) of a system without relying on a global clock. It is widely used for purposes such as checkpointing, debugging, deadlock detection, and termination detection.

Main Idea: Each process independently records its local state and the messages currently in transit on its incoming channels. When all local states and channel states are combined, they form a global snapshot corresponding to a consistent cut — a state that could have occurred during normal execution. Importantly, the snapshot is taken without stopping computation: processes continue their normal operation while the snapshot is collected.

Assumptions:

  • Communication channels are unidirectional and FIFO-ordered.
  • The network is strongly connected.
  • Communication is reliable and no process fails during snapshot collection.

Algorithm Steps:

  • Initiation: Any process can start the snapshot:
    • It records its local state.
    • It sends a special marker message on all its outgoing channels.
  • Receiving a Marker (at process Q from process P):
    • If Q has not yet recorded its state:
      • Record its local state.
      • Record the channel from P as empty.
      • Send markers on all outgoing channels.
      • Begin recording incoming messages on other channels until a marker arrives on each.
    • If Q has already recorded its state:
      • Messages received on that channel before the marker are considered in transit.
      • When the marker arrives, stop recording for that channel.
  • Termination: When a process has received a marker on all incoming channels, its local snapshot is complete. The collection of all local states and channel states represents a consistent global snapshot.

Termination detection

1What is the problem of termination detection in a distributed system and how can we solve it?

A distributed computation is said to have terminated when the following two conditions hold simultaneously:

  • All processes are idle1 (i.e., not performing any computation or waiting for events).
  • There are no messages in transit across any communication channel.

Intuitively, if all processes are inactive and there are no pending messages that could reactivate them, the entire system has stopped making progress.

Detecting this condition is difficult because each process has only local information — it knows its own state but not whether messages are still in transit between others, so processes must collaborate to detect global termination.

1. Termination Detection Using a Distributed Snapshot

The Chandy–Lamport snapshot algorithm can be adapted to detect termination. It records a consistent global state of the system and can verify whether all processes are idle and whether all communication channels are empty at the moment of the snapshot.

Implementation sketch (Tanenbaum's variant):

  • Each process defines a predecessor — the process from which it first received the snapshot marker — and a set of successors (those to which it sent markers).
  • When a process completes its part of the snapshot, it sends a message to its predecessor:
    • It sends a DONE message if:
      • All successors have already sent DONE messages, and
      • The process has received no new messages between recording its state and receiving the marker on each incoming channel.
    • Otherwise, it sends a CONTINUE message.
  • The initiator concludes that the computation has terminated only if it receives DONE messages from all successors.

Why This Does Not Always Work: The snapshot-based approach may fail because the predecessor relationship is not globally consistent. A process considers the first process from which it receives a marker as its predecessor, but this does not guarantee that it is also the predecessor of all its successors. As a result, the structure formed by predecessor/successor links may not cover the entire communication graph — meaning the termination detection may be incomplete or inconsistent.

2. Dijkstra–Scholten Algorithm for Diffusing Computations

The Dijkstra–Scholten algorithm (1980) is a specialized and efficient solution for detecting termination in diffusing computations — a class of computations that spread activity through message passing.

Assumptions:

  • All processes are initially idle, except for one initiator process.
  • A process becomes active only upon receiving a message.
  • Communication channels are reliable and FIFO.

Main Mechanism: The algorithm dynamically builds a spanning tree of active processes rooted at the initiator:

  • When an active process Pi sends a message to another process Pj:
    • If Pj was idle, it becomes active and Pi becomes its parent in the tree.
    • If Pj was already active, no new parent–child relation is established.
  • Each process maintains a counter of the number of outstanding messages it has sent but not yet acknowledged by its children.
  • When a process becomes idle (has no local work and no unacknowledged messages), it sends an ACK message to its parent.
  • The parent decrements its counter upon receiving each acknowledgment.
  • The computation terminates when the root (initiator) process becomes idle and has received acknowledgments from all its children.

Properties and Advantages:

  • No process outside the tree participates, minimizing message overhead.
  • Detection is accurate and complete — no false positives.
  • Works well for computations that start from a single initiator and spread outward (diffusing style).

Distributed transactions

Ch. 3 · Synchronization and logical time•p. 39•Question 14 of 32

Related exam questions · tap for a complete answer

1Describe pessimistic timestamp ordering and its alternatives.

A distributed transaction is a transaction that involves operations executed on multiple nodes or databases in a distributed system. Its goal is to maintain the ACID properties (Atomicity, Consistency, Isolation, Durability) across different sites, ensuring that all replicas or partitions behave as if the system were a single, centralized database.

To achieve this, a concurrency control protocol must ensure that concurrent transactions produce a result equivalent to some serial execution — this property is called serializability. Several techniques can guarantee serializability, including locking, timestamp ordering, and optimistic approaches.

Pessimistic Timestamp Ordering

The pessimistic timestamp ordering (TO) protocol ensures serializability by assigning each transaction T a globally unique timestamp TS(T) when it starts. This timestamp defines the transaction's logical position in the global serial order: all operations must respect this ordering.

Basic idea: Each data item X maintains two timestamps:

  • R_TS(X): timestamp of the last transaction that read X;
  • W_TS(X): timestamp of the last transaction that wrote X.

When a transaction issues a read or write, the system checks whether performing it would violate the global timestamp order:

  • Read rule: A transaction T with TS(T) can read X only if TS(T) ≥ W_TS(X). Otherwise, T would be reading an outdated value and is aborted.
  • Write rule: T can write X only if TS(T) ≥ R_TS(X) and TS(T) ≥ W_TS(X). Otherwise, the write is inconsistent with newer reads or writes, and T is aborted.

This approach is called pessimistic because conflicts are resolved immediately at operation time1 (by aborting conflicting transactions), rather than being postponed to commit time.

Example:

  • T1 (timestamp 10) reads X.
  • T2 (timestamp 20) writes X.
  • T1 later tries to write X.

Since T1 has an earlier timestamp and tries to overwrite a newer value written by T2, it must be aborted.

This protocol is called pessimistic because it checks for conflicts as soon as an operation occurs and aborts the transaction immediately if the timestamp order would be violated — preventing any inconsistent state from forming.

Alternative Approaches

1. Two-Phase Locking (2PL and Strict 2PL): In Two-Phase Locking, transactions acquire locks on data before reading or writing and release them only after finishing all operations. The protocol has two phases:

  • A growing phase, where locks are acquired and none are released.
  • A shrinking phase, where locks are released and no new locks are obtained.

This ensures serializability but can introduce deadlocks, since transactions might wait indefinitely for each other's locks.

A stricter version, Strict 2PL, holds all locks until the transaction either commits or aborts, guaranteeing not only serializability but also recoverability — no transaction can read or overwrite data written by an uncommitted transaction. Strict 2PL is widely used in real database systems because it simplifies recovery through rollback.

2. Optimistic Timestamp Ordering (Optimistic Concurrency Control): In contrast to pessimistic TO, optimistic timestamp ordering assumes that conflicts are rare. Transactions execute freely without coordination, maintaining a local log of read and write operations. Only at commit time does the system check for conflicts using timestamps or version numbers.

If no conflicts are found, the transaction commits; otherwise, it is aborted and restarted. This strategy eliminates blocking and deadlocks but may cause frequent aborts in high-contention environments.

Optimistic TO performs best in read-dominated workloads or systems with minimal concurrent conflicts, such as analytical or lightly loaded distributed databases.

3.9

Distributed deadlocks

Ch. 3 · Synchronization and logical time•p. 41•Question 15 of 32

A deadlock occurs in a distributed system when a set of processes are waiting for each other in a circular chain, such that none can proceed. In distributed systems, deadlocks are more complex because there is no global view and processes and resources may be located at different nodes.

Deadlock Detection: The Chandy–Misra–Haas Algorithm

The Chandy–Misra–Haas (CMH) algorithm is a fully distributed technique for detecting deadlocks without relying on a central coordinator. Each process cooperates by sending probe messages of the form PROBE(initiator, sender, receiver) along the wait-for dependencies. When a process Pi waits for a resource held by Pj, it sends PROBE(i, i, j); if Pj is also waiting for Pk, it forwards the probe as PROBE(i, j, k). If the probe eventually returns to its initiator Pi, a cycle is detected, meaning a deadlock exists. Then the initiator commits suicide or picks the process with the higher identifier and kills it.

The algorithm assumes reliable message delivery and static resource requests during detection. It is efficient and decentralized, allowing each process to independently detect deadlocks that involve itself.

Deadlock Prevention

Prevention techniques ensure that circular wait conditions cannot arise. A common approach is to assign each process or transaction a unique, globally ordered timestamp (for example, using Lamport clocks). Two well-known timestamp-based prevention strategies are:

  • Wait-Die: If an older transaction (with a smaller timestamp) requests a resource held by a younger one, it is allowed to wait. If a younger transaction requests a resource held by an older one, it is immediately aborted ("dies") and restarted later with the same timestamp. This rule ensures that only older transactions can wait for younger ones, preventing cycles.
  • Wound-Wait: When an older transaction requests a resource held by a younger one, it "wounds" the younger by preempting and aborting it, taking over the resource. If a younger transaction requests a resource held by an older one, it waits. This approach reduces waiting time for older transactions and, like Wait-Die, guarantees acyclic waits.

Both protocols prevent deadlocks by enforcing a total order of timestamps: the system never allows a younger transaction to block an older one, thereby eliminating the possibility of circular waiting.

Failure models in distributed systems

1Discuss (classify) the various types of failures that may happen to a distributed system and make a practical example of each of them.2Describe the failure model for a distributed system in general. In distributed systems a failure occurs when a system is no longer able to provide its expected services1. A failure results from an error, which in turn is caused by a fault. The ability of a system to continue providing its service despite the occurrence of faults is called fault tolerance.

A failure model defines and classifies the possible ways in which failures can manifest in the system. The most common classification distinguishes between:

  • Omission failures
  • Timing failures
  • Byzantine (arbitrary) failures

1. Omission failures

Omission failures occur when a component (process or communication channel) omits to perform an expected action, such as sending or receiving a message.

Process omissions. A process may fail to respond or stop executing entirely.

  • Fail-stop failure: the process halts completely and other processes can detect its failure with certainty. Example: a database server that crashes and reports its crash to the cluster manager.
  • Fail-silent (crash) failure: the process stops without any detectable signal; other processes cannot distinguish between a crash and a slow process. Example: a node in an asynchronous network that stops responding but might just be delayed.
  • Fail-safe failure: the process occasionally produces incorrect results, but these are easily recognizable. Example: a temperature sensor transmitting an incorrect value but with an invalid checksum.

Channel omissions. Channels can also experience omission failures, such as:

  • Send omission: the sender fails to insert a message into the channel (e.g., a software bug prevents message dispatch).
  • Channel omission: a message is lost during transmission (e.g., a dropped UDP packet due to network congestion).
  • Receive omission: the receiver fails to deliver an incoming message to the application (e.g., the operating system discards a message before processing).

In practice, many omission failures at the channel level are masked by transport protocols2 such as TCP, which employ acknowledgments and retransmissions to guarantee delivery.

2. Timing failures

Timing failures occur only in synchronous systems, where bounds on message delay and processing time are known. A timing failure happens when one of these bounds is violated.

  • Example: in a real-time distributed control system, a message is delivered too late, causing a safety violation.

In asynchronous systems, where message delays are unbounded, timing failures cannot be reliably detected.

3. Byzantine (arbitrary) failures

Byzantine failures represent the most general and difficult category. They occur when a component exhibits arbitrary or even malicious behavior—sending conflicting, incorrect, or inconsistent messages.

Process-level Byzantine failures. A process may:

  • send different messages to different peers,
  • perform incorrect computations,
  • or behave maliciously to disrupt consensus.

Example: in a replicated system, a faulty replica sends correct results to some clients and wrong results to others, preventing the group from agreeing on a single state.

Channel-level Byzantine failures. The channel itself may:

  • corrupt the content of a message,
  • duplicate or drop messages arbitrarily,
  • or deliver fictitious messages that were never sent.

Example: a corrupted network packet that is accepted by the receiver because of a failure in error detection.

In modern networks, Byzantine failures on channels are extremely rare thanks to error detection (checksums), retransmission, and authentication mechanisms. For this reason, Byzantine behavior is mostly considered only for processes, not for communication channels.

Most common failures

The most common process failure is the omission failure, where a process crashes or stops responding. In an asynchronous system, this is difficult to handle because there are no timing guarantees—one cannot tell whether a process is simply slow or has failed. As shown by Fischer, Lynch, and Paterson (1985), consensus is impossible even if a single process exhibits such behavior.

For communication channels, omission failures are less relevant since they are masked by reliable transport protocols (e.g., TCP) using retransmissions and acknowledgments. Instead, channels more often experience Byzantine-like failures, such as corrupted, duplicated, or out-of-order messages. These are easier to manage than Byzantine process failures because channels are passive: they cannot act maliciously, and simple techniques like checksums or message digests can detect and correct most errors.

The consensus problem is a fundamental challenge in distributed systems where a group of processes must agree on a single value1. For a consensus algorithm to be correct, it must satisfy three essential properties:

  • Agreement: No two correct (non-faulty) processes decide on different values. If process Pi decides on value vi and process Pj decides on vj, then vi = vj.
  • Validity: If all correct processes start with the same initial value v, then v is the only possible decision value. This prevents trivial solutions, like always deciding on a pre-determined constant.
  • Termination: All correct processes must eventually decide on a value in a finite amount of time. The system cannot remain undecided indefinitely.

Consensus is critical for tasks like electing a coordinator, committing distributed transactions, and ensuring replicated state machines remain consistent.

Feasibility of Consensus Under Failures

Achieving consensus is non-trivial, and its feasibility depends on the system model and the types of failures that can occur.

The FLP Impossibility Result

The celebrated FLP Theorem (Fischer, Lynch, and Paterson, 1985) establishes a fundamental limitation:

It is impossible for a deterministic algorithm to solve the consensus problem in a fully asynchronous distributed system if even a single process can fail by crashing.

The core issue is the inability to distinguish between a process that has crashed and one that is just experiencing extreme message delay. An algorithm might wait indefinitely for a response (violating Termination) or decide prematurely (risking a violation of Agreement).

Achieving Consensus in a Synchronous System

To overcome this impossibility, we can operate under a stronger set of assumptions, namely a synchronous system model.

In a synchronous system, timeouts can be used to reliably detect process crashes. If a process does not respond within the maximum time bound, it is considered to have failed.

The FloodSet Algorithm

The FloodSet algorithm is a protocol designed to solve consensus in a synchronous system that can tolerate up to f crash failures. Assumption that we make are that the system is is synchronous and that communication channels are reliable. The algorithm requires at least f + 12 rounds of communication to guarantee correctness.

Algorithm Description The algorithm works by ensuring that all correct processes converge to the same set of proposed values.

  • Initialization: Each process pi maintains a set Wi, initialized with its own starting value, {vi}. A global default value, v0, is also known to all.
  • Communication Rounds: For exactly f + 1 rounds, every process performs the following two steps:
    • Broadcast: Send its current set Wi to all other processes.
    • Update: Update its set by taking the union of its current set with all the sets it received in that round: Wi ← Wi ∪ (∪j Wreceived from j).
  • Decision: After f + 1 rounds are complete, each process pi makes its decision based on the final state of its set Wi:
    • If |Wi| = 1, decide on the single value in the set.
    • If |Wi| > 1, decide on the default value v0.

Proof of Correctness Rationale The correctness of the FloodSet algorithm, specifically its ability to guarantee the Agreement property, relies on a key lemma. This lemma demonstrates that despite up to f crash failures, all correct (non-faulty) processes will finish the algorithm with identical sets of values, leading them to make the same final decision.

The proof is based on the following chain of reasoning:

  • Synchronization in a Failure-Free Round: If a round of communication occurs where no process fails, all processes that are active at the end of that round will have identical sets of values (W). This round acts as a synchronization point. Thus, it is required at least one non-faulty process3 (k + 1 total processes).
  • Preservation of Synchronization: Once the sets of all active processes are identical, they will remain identical in all subsequent rounds. This is because each active process starts the next round with the same set, broadcasts the same set, and receives the same collection of messages from other active processes, thus performing the same union operation.
  • Conclusion after f + 1 Rounds: If two processes, i and j, are both active after all f + 1 rounds have completed, their final sets, Wi and Wj, must be identical.

Consensus Under Byzantine Failures

The Byzantine Generals algorithm is a solution to the consensus problem in a synchronous system where up to f processes can exhibit Byzantine failures (i.e., behave maliciously and unpredictably).

Imagine several army generals who must agree on a common plan (e.g., "attack" or "retreat"). They can only communicate via messengers. The challenge is that some generals may be traitors who will try to deceive the loyal generals by sending conflicting messages to disrupt their agreement. The goal is to ensure all loyal generals decide on and execute the same plan.

A solution is only possible if the total number of processes, N, is strictly greater than three times the number of faulty processes, f.

N ≥ 3f + 1

This means that over two-thirds of the processes must be correct. For instance, to tolerate one traitor (f = 1), at least four generals (N = 4) are required.

Algorithm Steps (for f = 1) The algorithm works in rounds, allowing loyal processes to identify and disregard conflicting information from traitors. Let's consider the case of one commander sending an order to several lieutenants.

  • Commander Sends Order: The commander sends their order (e.g., "attack") to every lieutenant.
  • Lieutenants Relay Orders: Upon receiving the order, each lieutenant relays the message they received to all other lieutenants. For example, Lieutenant A tells Lieutenant C, "The commander told me to attack."
  • Final Decision: Each lieutenant makes their final decision based on a majority vote of all the messages they have collected (the direct order from the commander and all the relayed orders from other lieutenants). If no majority exists, a default plan (e.g., "retreat") is chosen.

How the Algorithm Achieves Consensus The relaying step is the key to defeating the traitors.

  • Case 1: The Commander is Loyal. A loyal commander sends the same order to everyone. A traitorous lieutenant might relay a false order, but the other loyal lieutenants will relay the correct order. In each loyal lieutenant's final tally, the correct order will win the majority vote.
  • Case 2: The Commander is a Traitor. A traitorous commander might send different orders to different lieutenants (e.g., "attack" to L1, "retreat" to L2). When the loyal lieutenants relay these messages, they will expose the commander's deceit. Each loyal lieutenant will end up with a set of conflicting orders with no clear majority. They will all independently fall back to the same pre-agreed default plan, thus maintaining agreement.

By using this process of relaying and voting, the algorithm ensures that all loyal processes can converge on a single, consistent decision, even in the presence of malicious actors.

Reliable group communication, or reliable multicast, aims to guarantee that a message sent to a group of processes is successfully delivered to all its members. The complexity and implementation of these protocols depend heavily on the assumptions made about the reliability of processes and communication links.

Reliable Multicast with Non-Faulty Processes

In the simplest scenario, we assume that processes are reliable (non-faulty), but the communication channels are not, meaning messages can be lost2 (omission failures). The goal is to ensure that a message sent to a group is received by all of its members.

Basic Feedback-Based Approaches The fundamental way to achieve reliability over unreliable channels is to use a feedback mechanism combined with retransmissions1. However, naive implementations do not scale well.

  • Positive Acknowledgments (ACKs): In this approach, the sender multicasts a message and waits for an ACK from every single recipient. If a timeout expires before all ACKs are received, the sender retransmits the message. While simple, this method suffers from a critical issue known as ACK implosion, where the sender is overwhelmed by a massive number of acknowledgment messages from the group members.
  • Negative Acknowledgments (NACKs): To reduce feedback traffic, this optimistic approach requires receivers to send a NACK only when they detect that a message is missing. The sender must cache messages in a history buffer to handle retransmission requests. This significantly reduces the number of messages in normal operation but does not fully prevent feedback implosion if many receivers lose the same message, and it requires a mechanism for receivers to detect lost messages.

Scalable Feedback Control Approaches To overcome the limitations of basic feedback, scalable reliable multicast protocols focus on managing and reducing the amount of feedback traffic, especially in large groups.

  • Scalable Reliable Multicast - SRM: This approach uses a NACK-based strategy with a feedback suppression3 mechanism.
    • When a receiver detects a lost message, it schedules a NACK to be multicast to the entire group after a random delay.
    • If, during this delay, the receiver hears another NACK (or the retransmitted message itself) from another member for the same missing message, it suppresses its own NACK.
    • This ensures that, ideally, only one NACK is sent for any given lost message, preventing NACK implosion. However, a downside is that the multicast NACKs interrupt all processes in the group, even those that did not lose the message.
  • Hierarchical Feedback Control: This approach organizes the group members into a tree-like hierarchy.
    • The group is divided into smaller subgroups, each with a designated coordinator.
    • Feedback (such as NACKs) is sent only to the local subgroup coordinator. If the coordinator cannot fulfill a retransmission request (e.g., because it also missed the message), it forwards the request up the tree to its parent coordinator.
    • This structure dramatically improves scalability by localizing feedback and preventing the primary sender from being a bottleneck. Its main challenge is the complexity of building and maintaining the hierarchy.

The Challenge of Process Failures: Atomic Multicast

The problem changes fundamentally when we allow processes themselves to fail (e.g., by crashing). It is no longer sufficient to just guarantee eventual delivery to all members. The new goal becomes achieving atomic multicast, which provides stronger "all-or-nothing4" guarantees.

The Shift in the Problem When processes can fail, two critical issues arise:

  • Agreement on Delivery: If a process multicasts a message and then crashes, some members might have received it while others have not. We need to ensure that the message is either delivered to all surviving correct processes or to none of them.
  • Ordering: To maintain consistency in replicated systems, all correct processes must deliver messages in the exact same global order. This is known as Total Ordering.

These requirements define atomic multicast, which is a much stronger guarantee than simple reliable multicast.

The Solution: Virtual Synchrony The standard model for implementing atomic multicast in the presence of failures is Virtual Synchrony. It provides the illusion that all multicasts happen atomically5 with respect to process failures and group membership changes.

  • Group Views: The system's operation is divided into epochs, each defined by a group view, which is the set of currently active members. A view change occurs whenever a process joins, leaves, or is suspected of having failed.
  • View Changes as Barriers: A view change acts as a synchronization barrier. No multicast can cross from one view to another. The core guarantee is: if a message m is sent in a view v, it will either be delivered to all correct members of v before the new view v′ is installed, or it will be delivered to none of them.
  • Implementation Sketch: To achieve this, when a view change is initiated, processes stop sending new application messages. They first exchange any unstable messages (messages they have received but are not yet sure all others have) among the members of the old view. Once all unstable messages are flushed and acknowledged, they install the new view and resume communication.

By using virtual synchrony, the system can handle process failures gracefully, ensuring that the state of the application remains consistent across all surviving replicas. Moreover, different ordering guarantees can be combined with this reliable delivery to meet specific application needs. These orderings are enforced within each group view:

  • Reliable FIFO Ordering: Guarantees that messages from the same sender are delivered in the order they were sent.
  • Reliable Causal Ordering: Ensures that if a message m causally precedes another message m′, then m is delivered before m′ at all correct processes.
  • Reliable Total Ordering (Atomic Multicast): The strongest guarantee, ensuring that all messages are delivered to all processes in the exact same global order.

The Two-Phase Commit (2PC) and Three-Phase Commit (3PC) are atomic commit protocols designed to achieve agreement in distributed systems, particularly for distributed transactions. Their primary goal is to ensure atomicity (the "A" in ACID properties), guaranteeing that a transaction across multiple nodes either fully commits on all nodes or fully aborts on all nodes1, even in the presence of failures.

The Two-Phase Commit (2PC) Protocol

The Two-Phase Commit (2PC) protocol operates in an asynchronous system where servers can experience crash failures and messages can be lost, but it does not handle Byzantine (arbitrary) failures.

Functioning The protocol is coordinated by a single Coordinator and executed by multiple Participants. The process unfolds as follows:

  • The Coordinator sends an INIT message to start the transaction.
  • Participants, upon completing their work, send a COMMIT_REQUEST to the Coordinator.
  • Phase 1 (Voting): The Coordinator sends a VOTE_REQUEST to all Participants and enters a WAIT state. Each Participant then decides:
    • If it can commit, it sends a VOTE-COMMIT and enters a prepared state, where it can no longer unilaterally abort.
    • If it cannot commit, it sends a VOTE-ABORT.
  • Phase 2 (Decision): The Coordinator collects the votes.
    • If it receives a VOTE-COMMIT from all Participants, it sends a GLOBAL_COMMIT.
    • If even one Participant sends a VOTE-ABORT (or a timeout occurs), the Coordinator sends a GLOBAL_ABORT to all Participants.

Dealing with Possible Failures While 2PC is designed to handle failures, its main weakness is that it is a blocking protocol.

  • Participant Failure: If a Participant fails before voting, the Coordinator will time out and abort the transaction. If a Participant crashes after voting to commit, it will use its persistent log upon recovery to determine the final outcome from the Coordinator.
  • Coordinator Failure: This is the most critical failure scenario.
    • Failure before the vote: If the Coordinator crashes before sending the VOTE_REQUEST, Participants will time out and can safely abort.
    • Failure after the vote (The Blocking Problem): If the Coordinator crashes after Participants have voted VOTE-COMMIT but before it has broadcasted the final decision, the Participants are left in an uncertain or in-doubt state. They cannot unilaterally decide whether to commit or abort and must block, holding onto resources (like database locks) until the Coordinator recovers. While they can ask other Participants for the outcome, if all are in the same uncertain state, the system remains blocked.
    • Recovery: A recovering Coordinator uses its persistent log to determine the status of in-doubt transactions and re-sends the final decision to ensure all Participants reach a consistent state.

Correctness and Hypotheses

  • Safety: 2PC is considered safe because it never leads to an inconsistent state (i.e., it guarantees that all correct nodes will reach the same decision).
  • Liveness: The protocol sacrifices liveness because of its blocking nature. The indefinite wait caused by a Coordinator failure in the second phase is its defining drawback.

The Three-Phase Commit (3PC) Protocol

The Three-Phase Commit (3PC) protocol is an atomic commit protocol designed to be a non-blocking alternative to the Two-Phase Commit (2PC). Its primary goal is to prevent Participants from getting indefinitely stuck in an uncertain state, especially when the Coordinator fails.

Functioning 3PC introduces an intermediate "Pre-Commit" phase to ensure that all Participants are aware of the impending decision before the final commit is executed.

  • Phase 1 (Voting): This phase is identical to 2PC. The Coordinator sends a VOTE_REQUEST to all Participants. Participants respond with either VOTE-COMMIT if they are ready or VOTE-ABORT if they cannot proceed. The Coordinator waits for all votes.
  • Phase 2 (Pre-Commit): If the Coordinator receives a VOTE-COMMIT from all Participants, it enters a pre-commit state. It then sends a PREPARE_COMMIT message to all Participants. This message informs them that all nodes have agreed to commit. Upon receiving this, Participants acknowledge and move to their own pre-commit state.
  • Phase 3 (Decision): Once the Coordinator has received acknowledgments from all Participants for the pre-commit phase, it sends the final GLOBAL_COMMIT message.

Dealing with Possible Failures The main advantage of 3PC is its improved handling of failures, which makes it non-blocking under certain assumptions.

  • Participant Failure:
    • If a Participant fails to vote, the Coordinator times out and aborts the transaction for everyone.
    • If a Participant fails after the Coordinator has entered the Pre-commit state, the Coordinator can still proceed to a global commit, knowing that all other Participants are aware of the decision. The failed Participant can determine the outcome from others upon recovery.
  • Coordinator Failure: This is where 3PC's non-blocking nature shines. If the Coordinator fails, the remaining Participants can communicate among themselves to reach a decision without waiting for the Coordinator to recover.
    • If any other Participant has already aborted or is in the Init state, the decision is to abort.
    • If any other Participant has already committed, the decision is to commit.
    • If at least one Participant is in the Pre-commit state and a majority of nodes are in either Pre-commit or Ready, the decision is to commit. The pre-commit message acted as a guarantee that a global abort was no longer possible.
    • If a majority of Participants are in the Ready state but none have reached Pre-commit, the decision is to abort. This is safe because it's guaranteed that no Participant could have committed yet.

A crucial invariant that makes this possible is that it's impossible for one Participant to be in the Init state while another is in the Pre-commit state for the same transaction.

Correctness and Hypotheses

  • Non-Blocking Goal (Liveness): 3PC is designed to be non-blocking. The Pre-Commit phase ensures there is no single state that could lead to either a commit or an abort, thus eliminating the "uncertain" state of 2PC.
  • System Assumptions: For its non-blocking property to hold reliably, 3PC theoretically requires a synchronous system where network delays are bounded and failures are reliably detectable via timeouts.
  • Practical Limitations: In real-world asynchronous systems, where network partitions can occur, 3PC cannot guarantee both atomicity and liveness. Due to this limitation and its higher communication overhead (three message rounds vs. two), 3PC is not widely used in commercial systems, which often prefer the simpler, albeit blocking, 2PC protocol.

To write a reliable data store that can tolerate k faults, the number of replicas (processes) required depends directly on the type of failures you assume the system will experience.

Tolerance for Fail-Silent (Crash) Failures

If you assume that processes can only experience fail-silent or crash failures (where they simply stop working without sending incorrect data), you need a total of:

N = k + 1

processes to guarantee k-fault tolerance.

Why k + 1 is sufficient With crash failures, the only risk is that a replica becomes unavailable. The failed processes do not produce malicious or incorrect data. Therefore, if you start with k + 1 replicas and up to k of them crash, you are guaranteed to have at least one correct process1 remaining. This single surviving replica is sufficient to continue providing the service correctly.

Tolerance for Byzantine Failures

If you assume that processes can experience Byzantine failures (where they can behave arbitrarily, sending malicious or conflicting information), the requirements are much stricter. You need at least:

N ≥ 3k + 1

processes to guarantee k-fault tolerance in the general case (i.e., to solve the Byzantine Generals Problem). A simpler majority voting system would require at least N = 2k + 1.

Why 3k + 1 is necessary With Byzantine failures, faulty processes (or "traitors") can actively lie to disrupt agreement. The system must rely on a voting mechanism where the correct processes can outvote the faulty ones.

  • To ensure that the loyal processes can reach a majority even if all k traitors collude, the number of loyal processes (N − k) must be greater than the number of traitors (k). This gives the condition N − k > k, or N > 2k. The minimum integer N satisfying this is 2k + 1.
  • However, for full consensus where traitors can send conflicting information to different loyal processes, a simple majority is not enough. The stronger condition of N ≥ 3k + 1 is required to ensure that the loyal processes form a supermajority (more than two-thirds of the total) and can reach a definitive agreement by filtering out the lies.

Replicated State Machines (RSM)

An RSM is a fundamental abstraction in distributed systems that allows a collection of servers to work together as a coherent, fault-tolerant group and provide a continuous service.

Core Principles

  • State and Commands: A state machine manages an internal state (e.g., variables in a storage system) and processes external commands (requests, e.g., read/write operations).
  • Replicated Log: The core component ensuring consistency is the replicated log. It stores the sequence of commands. By guaranteeing that all replicas start with the same initial state and execute the same sequence of commands in the same deterministic order, all replicas maintain identical states (the State Machine Replication principle).
  • Fault Tolerance: Clients view the group of machines as a single, fault-tolerant entity.

Practical Context

RSMs are used for general consensus and continuous service in many applications. For instance, distributed databases utilize RSMs to ensure that individual partitions of a database do not fail, acting as a single entity managed by a set of N machines (e.g., in a group of N machines, they can continue operating if a minority, N − 1, fails simultaneously).

Raft: An Understandable Consensus Algorithm

Raft is a consensus algorithm that is functionally equivalent to multi-Paxos in terms of assumptions, guarantees, and performance, but designed primarily to be easily understood, explained, and implemented. It achieves total order broadcast, which is key to RSM implementation.

Failure Model and Guarantees

Raft operates under a realistic set of assumptions common in distributed environments:

  • Communication: Unreliable and asynchronous (messages can be lost, duplicated, or arbitrarily delayed — no byzantine failures).
  • Processes: Can crash and restart, requiring state to be stored on durable storage to remember past progress.

Raft provides two main guarantees:

  • Safety (Correctness): All non-failing machines execute the same command in the same order. Safety is always maintained.
  • Liveness / Availability: The system makes progress if a majority (quorum) of machines are up and communicating. While not deterministically guaranteed in theory (FLP theorem), it is ensured in practice under normal operating conditions through mechanisms like randomization.

Raft's Problem Decomposition

Raft splits the consensus problem into three distinct subproblems for clarity:

  • Leader Election: A single server is chosen to act as the coordinating Leader.
    • Server States: Follower, Candidate, and Leader. All nodes start as Followers.
    • Terms: Time is divided into sequential terms. A Follower starts an election by becoming a Candidate if its randomized heartbeat timeout expires.
    • Quorum: A Candidate wins by receiving votes from a majority of servers (a quorum). The Leader then sends periodic AppendEntries messages (heartbeats) to maintain its authority.
  • Log Replication: The process by which the Leader disseminates client commands and maintains log consistency.
    • Process: Client commands are routed to the Leader, which appends them to its log and replicates them to Followers via AppendEntries RPCs.
    • Commit: An entry is committed once it is replicated on a majority of servers. The Leader then executes the command on its state machine and notifies the Followers to do the same.
    • Log Matching: Raft guarantees that if log entries on different servers share the same index and term, they are identical, and all preceding entries are identical. The Leader repairs inconsistencies automatically during normal operation by rejecting mismatched entries and retrying with prior ones.
  • Safety: Rules ensuring the system remains correct, especially during elections.
    • Up-to-Date Log: Only servers with an up-to-date log can become Leader. A voting server will deny its vote to any candidate whose log is considered less complete than its own.
    • Client Communication: Clients always interact with the Leader. Raft provides linearizable semantics (operations execute once, in order). Clients use sequential identifiers to handle retries and enable servers to discard duplicate commands.

Consistency models: properties, availability, and evolution

A consistency model is a contract between processes and the distributed data store. It describes what a process can expect when reading or updating shared data, knowing that other processes are accessing the same data.

Since it's impossible to determine which write operation was truly the "last" without a perfectly synchronized global clock, various models offering differing guarantees have been developed:

  • Stricter Guarantees: (e.g., sequential or linearizability). These simplify development but incur high costs.
  • Weaker Guarantees: These reduce implementation costs (especially synchronization overhead), but make development harder because the programmer must explicitly handle potential inconsistencies.

There are several models for individual operations or groups of operations:

  • Guarantees on content
    • Maximum "difference" on the versions stored at different replicas.
  • Guarantees on staleness
    • Maximum time between a change and its propagation to all replicas.
  • Guarantees on the order of updates
    • Constrain the possible behaviors in the case of conflicts.
    • Data-centric vs client-centric.

Consistency protocols

A consistency protocol implements consistency models. There are different implementation strategies. The primary strategies can be categorized based on several architectural choices.

Passive vs. Active Replication

In a passive replication scheme, a single replica, known as the master (or primary), handles all incoming operations. The master processes the request, updates its state, and then propagates these changes to one or more backup replicas (slaves). If the master fails, a failover process is initiated where one of the backups is elected as the new master. Fault tolerance is effectively achieved only if the propagation of changes is synchronous, ensuring backups are up-to-date before confirming an operation. A key limitation of this model is the lack of workload sharing, as only the master is active in processing requests.

To overcome the limitations of the passive model, active replication allows all replicas to process user requests1 directly. This approach enables workload sharing, but it requires more complex coordination to ensure that all replicas process the same operations in the same order to maintain a consistent state across the system.

Replication Architectures

In a Single Leader model, one replica is designated as the leader.

  • Writes: All write requests are sent exclusively to the leader. The leader applies the write to its local storage first and then sends the update to all other replicas, known as followers.
  • Reads: Read requests can be handled by any replica, including the leader or any of the followers.

A Multi-Leader architecture allows multiple replicas to act as leaders, concurrently accepting write requests.

  • Writes: Clients can send write requests to different leaders simultaneously. Since there is no single entity ordering all writes, this can lead to write-write conflicts if the same data is modified concurrently on different leaders.
  • Conflict Resolution: The resolution of such conflicts depends on the specific consistency model being implemented.

In a Leaderless model, the client directly contacts several replicas2 to perform read and write operations.

  • Mechanism: This approach uses a voting-like system called quorums. A write operation is considered successful only after a majority of designated replicas agree on it. Similarly, an agreement among a quorum of replicas is required to return a value for a read operation.

Update Propagation Models

In syncronous protocols the write operation is considered complete only after the leader has received a confirmation (reply) from all of its followers. This ensures the highest level of consistency at the cost of higher latency.

In asynchronous ones, the write operation completes as soon as the new value is stored on the leader. The followers are updated in the background asynchronously. This model offers lower latency but weaker consistency guarantees.

Finally, in a semi-synchronous protocol this is a hybrid approach where the write operation completes once the leader has received a reply from a predefined number (k) of replicas. It offers a balance between the performance of asynchronous replication and the durability of synchronous replication.

Data-centric models

Data-Centric Consistency Models are fundamental in distributed systems that manage replicated data. They focus on providing a consistent view of the data at a system-wide level, even in the presence of concurrent access.

Strict Consistency

This model guarantees that any read on a data item x returns the value of the most recent write on x. All writes are instantaneously visible, and a global time order is maintained. This model is ideally achievable only in uniprocessor systems and is not feasible in distributed systems.

Linearizability

Linearizability (also known as strong, atomic, or external consistency) is the strongest possible consistency guarantee in the presence of replication.

Each operation (read or write) must appear to execute instantaneously at a specific moment in time, between the operation's start and end, along a single timeline. It provides the illusion that there is only one copy of the data and that all operations on it are atomic.

It implies a total ordering of all operations and guarantees causality, but is expensive to implement as it requires blocking and synchronous coordination protocols.

Sequential Consistency

The result of any execution is the same as if the read and write operations of all processes were executed in *some* sequential order, and the operations of each individual process appear in this sequence in the order specified by its program. Sequential consistency is weaker than linearizability as it does not include a notion of absolute real-time, but still all processes must see the same sequence of operations (the same interleaving).

Implementation

We consider two important assumptions:

  • FIFO Links: followers receive updates in the same order the leader sent them.
  • Sticky Clients: It is often assumed that clients always read from the same replica to maintain a consistent view from their perspective.

Figure: Two timelines for processes P1 and P2 (P1: W(x)a; P2 reading). The left is labelled "Consistent"; the right is labelled "NOT Consistent (sequential but not linearizable)". Caption: Not linearizable because P2's first read R(x)NIL begins after P1's write W(x)a has completed, yet it returns a stale value instead of 'a'. However, it is sequentially consistent because a valid global sequence of operations (R(x)NIL → W(x)a → R(x)a) can be constructed that respects the internal program order of each process.

Two primary strategies are used to implement Sequential Consistency:

  • Single-Leader Protocols: a designated leader acts as the central point for serializing all write operations. It must use synchronous or semi-synchronous replication to be fault-tolerant.
  • Leaderless (Quorum-Based) Protocols: clients interact with multiple replicas simultaneously. To prevent conflicts and ensure a globally consistent order, specific quorum conditions must be met:
    • NW > N/2: The write quorum (NW) must be a strict majority of the half of total replicas (N) to prevent write-write conflicts.
    • NR + NW > N: The sum of the read quorum (NR) and the write quorum must exceed the total number of replicas to prevent read-write conflicts, ensuring every read sees the latest write.

Causal Consistency

Causal consistency weakens sequential consistency by distinguishing between operations that are potentially causally related (according to Lamport's happens-before relation) and those that are not (concurrent). Writes that are potentially causally related must be seen by all processes in the same order. Concurrent writes may be seen in any order on different nodes.

Caption: The bottom example is NOT causally consistent because P2's R(x)a establishes a causal dependency where W(x)a → W(x)b. Implementation

Multi-leader implementations are possible (which enable concurrent updates). Writes are timestamped with Lamport's vector clocks to inform the system about what the process knows about other processes. In this way a process knows the causality relations between operations. An update U is applied to a replica only when all the write operations that are possible causes of U have been received and applied, otherwise a read always returns the previous value.

FIFO Consistency (PRAM)

FIFO consistency (also known as PRAM Consistency) is weaker than causal consistency as it ignores causality between different processes. Writes performed by a single process are seen by all others in the order in which they were issued; writes from different processes may be seen in any order on different nodes (example in Fig. 5.4).

Eventual Consistency

This is the weakest consistency model and is the foundation for many non-relational (NoSQL) systems. It is often sufficient in systems where there are no simultaneous updates or they can be easily resolved, and operations are predominantly reads (e.g., Web caches, DNS, social media). In the absence of new write operations for an indefinite period, all replicas will eventually converge to the same identical copy of the data.

This model is very popular since it is easy to implement, very few conflicts happens in practice and it uses dedicated data types (Conflict-free replicated data types). EC promises that replicas will converge eventually, but doesn't specify how. CRDTs provide a concrete mechanism that mathematically guarantees this convergence automatically, even under concurrent updates and arbitrary message ordering. Despite their advantages, CRDTs have limitations:

  • Limited Expressiveness (Semantic Dependency): They are not a universal solution for all data structures or arbitrary application logic.
  • No Guarantees Before Convergence: During the period before convergence is reached (i.e., while updates are still propagating or conflicts are being resolved), there are no guarantees about the state read from a replica.

Despite their advantages, CRDTs have limitations:

  • Data Model and Semantic Dependency: The automatic conflict resolution provided by CRDTs is specific to the data type and the semantics of its operations (e.g., commutativity). They are not a universal solution for all types of data or conflicts. Developers might still need to understand the underlying conflict handling for operations not covered by standard CRDTs.
  • External Side Effects: CRDTs are effective for managing replicated state but do not help manage external side effects of operations (e.g., sending an email or charging a credit card). Ensuring such actions happen exactly once requires different mechanisms.
  • Custom Logic for Complex Operations: For application-specific merge logic or operations that don't fit standard CRDT models, developers might still need to implement custom conflict resolution, potentially falling back to simpler, lossy strategies like Last-Write-Wins.
  • Lack of Total Ordering / Strong Constraints: CRDTs ensure eventual convergence but do not impose a total order on concurrent operations. This makes them unsuitable for scenarios requiring strict guarantees like uniqueness constraints (e.g., ensuring a username is unique across all replicas simultaneously). Such constraints typically require stronger consistency models involving consensus.

Consistency Models and High Availability

A protocol is considered Highly Available (HA) if it does not require synchronous, blocking communication, allowing a client to receive a response from a functioning replica even if other parts of the system are unavailable or partitioned.

Based on this, both Causal and FIFO (PRAM) consistency models are Highly Available. Causal consistency is the strongest HA model because it avoids synchronous coordination, often using vector clocks to manage dependencies without being slowed by network delays. FIFO consistency is also highly available and simple to implement, as it only requires preserving the order of writes from a single sender, which can be managed with sequence numbers.

Conversely, Linearizability and Sequential consistency are NOT Highly Available. Linearizability, being the strongest model, requires costly synchronous protocols3 to guarantee global real-time ordering, which inherently limits availability and increases latency. Although weaker, Sequential consistency also requires synchronous updates to ensure all processes see the same global order, causing the system to block during network partitions.

Client-Centric Consistency Models

Client-Centric Consistency Models were developed to address inconsistency issues perceived by a single client4 interacting with a distributed, replicated data store, especially in scenarios involving mobility or variable connectivity where the client may connect to different replicas over time.

The main problem they solve is what happens if a client dynamically changes the replica it connects to. If updates previously performed by the client have not yet been propagated to the new replica, the client would perceive inconsistent behavior, as if its previous changes had never occurred.

Client-centric models include four desirable properties:

Monotonic Reads

This property guarantees that if a process reads the value of a data item x, any subsequent read of the same data item by the same process will always return the same value or a more recent value.

Guarantee Once a process has read a value from a replica, it will never see an older (more stale) version of the data from a read performed at a different replica at a later time.

Monotonic Writes

This property states that a write operation performed by a process on a data item x is completed before any subsequent write operation on the same data item by the same process.

Guarantee If a client performs two consecutive writes, the second write will not take effect until the first one has been applied. This consistency is similar to FIFO consistency but is limited to the operations of a single process, ensuring that its writes are applied everywhere in the correct order.

Read Your Writes

This is a crucial guarantee for the user, often also called Read-after-Write Consistency.

Guarantee The effect of a write operation by a process on data item x will always be seen by a subsequent read operation on the same data item by the same process, regardless of the replica consulted. In other words, the write is always completed before a subsequent read by the same process (example in Fig. 5.7).

Writes Follow Reads

This property guarantees that if a process performs a write operation on data item x after having read it, that write is guaranteed to take place on the same or a more recent value of x than what was read.

Guarantee Any subsequent write operation is performed on a copy of the data that is up-to-date with respect to the most recently read value by that process. For example, this ensures that a user sees the posting of a reply to a newsgroup article only after having seen the original article.

Implementation of Client-Centric Models

To implement client-centric models, strategies are adopted to track the access history of each client:

  • Unique Identifier: Each operation receives a unique identifier (e.g., replica ID + sequence number).
  • Read-Set: For each client, a Read-Set is defined, containing the identifiers of the writes relevant to the read operations performed by the client.
  • Write-Set: A Write-Set is defined, containing the identifiers of the writes performed by the client.

These sets (which can be encoded as vector clocks) are used to enforce consistency when accessing a different replica:

  • Monotonic Reads: Before performing a read on replica L2, the client verifies that all writes in its Read-Set have also been applied on L2.
  • Monotonic Writes / Read Your Writes: Before a new write or read, the client ensures that the target replica has seen all writes present in the Write-Set.
  • Writes Follow Reads: The server replica is updated using the client's Read-Set before the new write is applied and added to the Write-Set.

To improve efficiency, client operations are often grouped into sessions. Sessions are opened when the application starts and closed upon exit, allowing the associated Read-Sets and Write-Sets to be cleared.

Replica Placement

Deciding where to place data replicas involves three primary strategies:

  • Permanent Replicas: A fixed set of statically configured core servers that form the backbone of the distributed system.
  • Server-Initiated Replicas: Replicas created dynamically by the system to handle high load or reduce latency by moving data closer to users.
  • Client-Initiated Replicas: Essentially client-side caching, where clients temporarily store local copies of data for faster access.

Update Propagation

Update propagation defines how changes are communicated between replicas.

What to Propagate There are three choices for the content of an update message:

  • Notification/Invalidation: Send only a notification that data is stale. This is efficient for write-heavy workloads.
  • Data Transfer: Send the actual modified data. This is best for read-heavy workloads.
  • Operation Propagation: Send the update operation itself for each replica to execute (active replication). This has low network overhead but requires deterministic operations.

How to Propagate Updates can be sent using two main methods:

  • Push-based: The server proactively sends updates to other replicas, which is suitable for maintaining high consistency.
  • Pull-based: Replicas or clients request updates on demand, ideal for client caches. Typically used to manage client caches.

  • Client C reads from Follower 2 before W(x)1 arrives, getting R(x)0.

Eventual consistency

1Does eventual consistency imply sequential consistency?2Does eventual consistency imply causal consistency?3Does eventual consistency imply FIFO consistency?4Consider a messaging system implemented on top of an eventually consistent datastore. How could you make the system FIFO consistent? How could you you make the system causally consistent?

No, eventual consistency (EC) does not imply sequential, causal, or FIFO consistency. EC is the weakest consistency model among these.

  • Sequential Consistency (SC): No. EC only guarantees that replicas will eventually converge1 to the same value if writes stop. It makes no guarantees about the order in which writes are observed across different processes or even by a single process over time. SC, conversely, demands that all operations appear to execute in some single total sequential order consistent with program order across all processes. EC allows temporary inconsistencies and orderings that violate SC.
  • Causal Consistency (CC): No. EC does not guarantee that causally related operations (e.g., a read following a write that influenced it, or a write following a read) are seen in their causal order by all processes. Due to replication lag, a system providing only EC might allow a process to observe an effect before its cause2, which violates CC. CC requires that the potential causal order (happens-before relationship) is respected everywhere.
  • FIFO Consistency: No. EC does not guarantee that writes from a single process are observed3 in the order they were issued. A client reading from different replicas in an eventually consistent system might observe older data after having already seen newer data (violating Monotonic Reads), which is incompatible with FIFO consistency. FIFO consistency specifically requires that writes from the same source are seen in their issuance order by all observers.

Messaging system

An eventually consistent (EC) datastore only guarantees eventual convergence, not specific ordering during propagation. To achieve stronger guarantees like FIFO or Causal consistency for a messaging system built on it, ordering mechanisms must be added4 at a higher layer (application or middleware).

Achieving FIFO Consistency We want to ensure messages from the same sender are delivered in the order they were sent. Messages from different senders can be interleaved arbitrarily. We can simply attach increasing sequence numbers to messages. At receiver side, messages arriving out of order are typically buffered in a hold-back queue until the missing predecessors arrive.

Achieving Causal Consistency We want to ensure messages that are causally related (based on the happens-before relationship →) are delivered in that order to all recipients. Concurrent messages can be delivered in different orders. We can use vector clocks.

Each process Pi maintains a vector clock Vi. When Pi sends a message, it increments its own entry Vi[i] and attaches the entire vector Vi as a timestamp (tm) to the message. This timestamp captures the sender's knowledge of events across the system, representing the message's causal past. A receiver Pj delays the delivery of a message m from sender Pi with timestamp tm until both conditions are met:

  • 1. Pj has delivered the immediately preceding message from Pi: tm[i] = Vj[i] + 1.
  • 2. Pj has delivered all messages that Pi knew about when it sent m: tm[k] ≤ Vj[k] for all k ≠ i.

Messages that fail these conditions are buffered until their causal dependencies are satisfied.

MapReduce is a programming model introduced by Google in 2004 for processing large-scale data across distributed clusters. It was designed to address the challenges of data-intensive applications where data volumes are measured in terabytes or more, far exceeding the capacity of a single machine.

The Execution Model: The MapReduce computation consists of three main phases:

  • 1. Map Phase: The map function processes individual input elements independently. For each element, it outputs one or more ⟨key, value⟩ pairs. Map tasks are stateless and can execute in parallel across multiple machines.
  • 2. Shuffle and Sort Phase: The MapReduce framework automatically groups all key-value pairs by their keys and ensures that all values with the same key are sent to the same reducer. This phase is managed entirely by the system.
  • 3. Reduce Phase: The reduce function receives a key and an iterator over all values associated with that key. It processes these values to produce one or more output values. Reducers for different keys can execute in parallel.

Architecture and Fault Tolerance: MapReduce operates in a master-worker architecture:

  • Input data is split into M map tasks (typically 64 MB blocks)
  • The reduce phase is partitioned into R reduce tasks using a hash function: hash(key) mod R
  • The master assigns tasks to workers dynamically, considering data locality to minimize network bandwidth usage
  • Worker failures are detected via periodic heartbeats, and failed tasks are automatically re-executed
  • Stragglers (slow tasks) are mitigated by scheduling redundant executions near job completion

Computing Average Product Ratings

Now we apply the MapReduce model to solve the specific problem of computing average product ratings from a dataset of tuples ⟨user, product, rate⟩. The goal is to aggregate all ratings (rate) by product identifier and compute the average rating for each product. To do so, we need both the sum of all ratings1 and the count of ratings for each product. Therefore, the map function will emit pairs containing both pieces of information.

Map Function: The map function processes each input record ⟨user, product, rate⟩ independently. It extracts the product as the key and emits a pair containing the rating and a count of 1.

// Input: <key, record> where record is <user, product, rate>
def map(record):
// Extract the product as the key
product = record.product

// Extract the rating
rate = record.rate

// Emit the tuple <product, [rate, 1]>
// [rate, 1] represents the rating value and count (always 1)
emit(product, [rate, 1])

Shuffle and Sort Phase: The MapReduce framework automatically groups all key-value pairs produced by the mappers with the same key (product) and routes them to the same reducer. For example, if product ProductA received ratings 4, 5, and 3 from different users, the reducer for ProductA will receive the list: [[4, 1], [5, 1], [3, 1]].

Reduce Function: The reduce function receives a product key and an iterator over all [rate, count] pairs associated with that product.

// Input: <key, values_list> where key is the product,
// values_list is an iterator over [rate, count] pairs
def reduce(product, values_list):
total_sum = 0
total_count = 0

// Iterate over the immutable list of values
for value in values_list:
// value is [rate, count]
rate = value[0]
count = value[1] // always 1 in this case

total_sum += rate
total_count += count

// Calculate the average (Average = Sum / Count)
average_rate = total_sum / total_count

// Emit the average rating for the product
emit(product, average_rate)

Advantages of the MapReduce Solution

This MapReduce implementation provides several key benefits:

  • Automatic parallelization: Map tasks process input records independently, and reduce tasks process different products in parallel, enabling massive scalability
  • Fault tolerance: The framework automatically detects node failures and re-executes failed tasks without manual intervention
  • Data locality: The MapReduce scheduler attempts to run map tasks on machines where the input data is stored, minimizing network bandwidth usage
  • Simplicity: Developers only need to write the map and reduce functions; all distributed system complexities are handled by the framework
  • Scalability: The solution scales horizontally to hundreds or thousands of machines without code changes

Limitations

While MapReduce is powerful for batch processing, it has some limitations:

  • High overhead: Each MapReduce step must complete before the next one can start, introducing latency
  • Fixed paradigm: The two-phase structure (map then reduce) is rigid and may require multiple chained jobs for complex computations
  • Lower raw performance: Compared to specialized high-performance computing solutions, MapReduce has more overhead

Modern dataflow systems like Apache Spark and Apache Flink extend the MapReduce model by supporting arbitrary acyclic graphs of transformations rather than just two phases, providing more flexibility and better performance for iterative algorithms.

The dataflow programming model represents a significant evolution beyond the traditional MapReduce paradigm.

While MapReduce restricts computation to only two processing stages (map and reduce), the dataflow model generalizes this concept to support arbitrary acyclic graphs1 (DAGs) of transformations. This architectural shift enables more complex data processing pipelines without requiring multiple sequential MapReduce jobs.

Core Characteristics: The dataflow model exhibits several fundamental characteristics that distinguish it from simpler processing paradigms:

  • Arbitrary DAG of transformations: Instead of being limited to two stages, dataflow programs can express computations as directed acyclic graphs with any number of transformation operators. Each node in the graph represents a data transformation operation, and edges represent data flow between operations.
  • Unified abstraction for batch and stream processing: The same programming model and execution engine can handle both batch processing (analyzing static datasets) and stream processing (continuously processing incoming data). This unification simplifies system architecture and allows code reuse.
  • Immutable data transformations: Following the functional programming paradigm, dataflow operations transform immutable input data into new outputs rather than modifying global mutable state. This approach simplifies reasoning about correctness and enables better parallelization.
  • Automatic parallelization: The runtime system automatically decomposes logical operations into parallel physical tasks that can execute concurrently across multiple machines. Developers specify what computations to perform, not how to distribute them.
  • Declarative programming interface: Developers write high-level descriptions of data transformations using familiar constructs like map, filter, reduce, join, and aggregate. The system handles all distributed computing complexities.
  • Fault tolerance: The framework provides automatic recovery from node failures through mechanisms like lineage tracking (recomputing lost data) or distributed snapshots. Applications do not need to implement their own failure handling logic.
  • Data locality optimization: The scheduler attempts to execute tasks close to where their input data resides, minimizing network bandwidth consumption. This is particularly important when processing terabytes of data across distributed storage systems.

Scheduling of Tasks

The scheduling approach, exemplified by Apache Spark, organizes execution into discrete stages that are scheduled and executed sequentially.

Execution Model:

  • The dataflow graph is divided into stages, where stage boundaries occur at operations requiring data redistribution (shuffles)
  • Tasks within each stage execute in parallel, processing different partitions of data
  • A stage must complete entirely before the next stage begins execution
  • Intermediate results between stages are materialized to storage (memory or disk)
  • The scheduler dynamically assigns tasks to workers at runtime, considering data locality and current cluster state

Advantages of Scheduling:

The scheduling architecture provides several benefits that stem from decoupling upstream and downstream stages:

  • 1. Runtime elasticity: Since scheduling decisions occur at runtime after observing the output of previous stages, the system can dynamically determine the optimal number of tasks and resources for each stage. If stage A produces more data than expected, stage B can be allocated additional resources to handle the load.
  • 2. Dynamic load balancing: The scheduler can examine the distribution of intermediate data and intelligently partition it across tasks in the next stage to ensure balanced workloads. This avoids scenarios where some tasks finish quickly while others become bottlenecks.
  • 3. Fine-grained fault tolerance: Because intermediate results are persisted to durable storage, recovery from failures is efficient. The system maintains the lineage (computational history) of each data partition. Upon failure, only the specific tasks necessary to recompute lost data need to be re-executed, rather than restarting the entire job.
  • 4. Memory caching for iterative algorithms: Intermediate datasets that are reused multiple times (common in machine learning algorithms) can be cached in memory across stages, significantly improving performance.

Disadvantages of Scheduling:

The scheduling approach introduces certain overheads:

  • Higher latency: The need to materialize intermediate results to storage and the scheduling overhead between stages introduces latency in the order of hundreds of milliseconds to seconds. This makes the approach less suitable for low-latency stream processing applications.
  • I/O overhead: Writing intermediate results to storage (even memory) and reading them back consumes resources and adds delays.

Pipelining of Tasks

The pipelining approach, exemplified by Apache Flink, instantiates all tasks at job submission2 and allows them to execute concurrently, forming a continuous processing pipeline.

Execution Model:

  • When a job is submitted, all tasks for all stages are immediately instantiated and deployed to worker nodes
  • Tasks communicate through persistent TCP channels that form a network topology matching the dataflow graph
  • As soon as a task has data available from its upstream tasks, it begins processing—it does not wait for upstream stages to complete
  • Data flows continuously through the pipeline without being materialized to external storage between stages
  • Operators from different logical stages execute simultaneously, overlapping their computation

Advantages of Pipelining:

The continuous execution model provides distinct benefits:

  • 1. Low latency: Since there is no scheduling overhead between stages and no need to write/read intermediate data to storage, data flows through the pipeline with minimal delay. This makes pipelining ideal for stream processing applications requiring subsecond latencies.
  • 2. Efficient resource utilization: All stages are active simultaneously, allowing the system to fully utilize CPU, memory, and network resources throughout execution. There are no idle periods waiting for scheduling decisions.
  • 3. Natural support for streaming: The continuous execution model naturally accommodates unbounded data streams, as operators can process data incrementally as it arrives.

Disadvantages of Pipelining:

The pipelining architecture has certain limitations:

  • No runtime elasticity: Resource allocation decisions are made at job submission time, before observing the data. The system cannot dynamically adjust the number of tasks or resources per stage based on actual data characteristics.
  • Limited dynamic load balancing: Without runtime scheduling, the system has limited ability to rebalance workloads if initial partitioning creates skewed loads.
  • Coarse-grained fault tolerance: Failure recovery is based on distributed snapshots (similar to the Chandy-Lamport algorithm). The system periodically captures global consistent snapshots by propagating marker messages through the pipeline. Upon failure, the entire job must be restarted from the last completed snapshot, and all data processed after that snapshot must be replayed. This is less efficient than the fine-grained lineage-based recovery in scheduling systems.

Fault Tolerance Mechanism in Pipelining:

The distributed snapshot protocol works as follows:

  • 1. Source operators periodically inject special marker messages into the data stream
  • 2. When a task receives a marker from an input channel, it stops processing data from that channel
  • 3. Once markers arrive from all input channels, the task:
    • Saves its current state (e.g., window contents, aggregation values) to durable storage
    • Propagates a marker downstream to its output channels
    • Resumes normal processing
  • 4. This process creates a globally consistent snapshot across all tasks
  • 5. Upon failure, the system redeploys all tasks, restores state from the last snapshot, and replays input data received after that snapshot

Stream Processing in Data Analytics

Stream processing is a programming paradigm for processing continuous data streams in real-time, particularly relevant in modern data analytics contexts. Unlike batch processing, where static datasets are analyzed to produce results, stream processing inverts this paradigm: from "static data and on-demand queries" to "static queries and dynamic data". A stream processing job continuously produces new results as new data arrives.

Fundamental Concepts Stream processing is based on two core concepts: data streams that flow through the system, and processing operators that transform the data. For example, computing the average temperature in the last hour for each room in a building, continuously updating the value as new sensor readings arrive.

Windows: The Solution for Continuous Queries

Windows provide precise semantics for continuous jobs. They isolate the portion of the stream on which a computation is performed and define when such computation should occur.

Types of Windows There are two main categories of windows:

Count-based (physical) windows are based on a fixed number of elements. They are defined by two parameters: size (the number of elements in the window) and slide (how many elements to advance). For example, a window with size = 4 and slide = 2 would contain 4 elements and advance by 2 elements at a time.

Time-based (logical) windows are based on a fixed time interval. For example, "a 1-hour window that slides every 5 minutes" would compute results over 60-minute intervals, updating every 5 minutes.

Window Classification Windows can be further classified based on their overlap behavior:

  • Sliding windows: elements overlap between consecutive windows (slide < size)
  • Tumbling windows: no overlap between windows (slide = size)

The Clock Problem

A fundamental question in stream processing is: which clock do we use to measure time? This leads to two distinct timing models, each with significant implications for correctness and performance.

Processing Time Processing time is the wall-clock time of the machine executing the computation. While it is easy to manage, it has critical disadvantages:

  • It is outside the application's control
  • There is no explicit synchronization between input and output; if the system is overloaded, data queues up
  • With multiple sources, the processing order depends on network delays between sources and the processor
  • With distributed processing, order also depends on the speed and load of each executor

Event Time Event time is a metadata (timestamp) associated with each data element. The timestamp is assigned by sources or by the ingestion system. This gives sources full control over the processing order. However, it introduces challenges:

  • The processor might receive elements out-of-order (with respect to their timestamps)
  • It must buffer elements until it is certain that no earlier elements will arrive
  • This may introduce undesired delays

Watermarks: Solving the Ordering Problem

Watermarks represent a contract between sources and the processor. They solve the out-of-order problem inherent in event time processing.

How Watermarks Work Periodically, sources produce watermarks containing an event time t. A watermark with time t indicates that the source will never again produce elements with time ≤ t. When the processor receives a watermark with time t from all sources, it can safely process all elements with event time ≤ t.

An important property is that order is preserved even in the presence of distributed processing. When a processor receives a watermark with time t from all input channels, it propagates the watermark downstream.

Implementation Approaches

There are two main architectural approaches for implementing stream processing systems, each with different trade-offs between latency, throughput, and resource management.

Pipelining (Apache Flink) In the pipelining approach, operators are always active and store necessary state locally. All tasks are instantiated as soon as the job is submitted, and they communicate via TCP channels. A task can begin processing as soon as data is available, creating a pipeline where operators from different stages can execute simultaneously.

Fault Tolerance: Based on distributed snapshots (similar to the Chandy-Lamport algorithm).

Scheduling/Micro-batch (Apache Spark) In the scheduling approach, the system divides input data into micro-batches corresponding to the size/slide of each window. A different job is executed for each micro-batch. The window state after each micro-batch is saved to external storage and recovered in the next micro-batch.

Fault Tolerance: Easy and fine-grained, based on data lineage. Only the tasks necessary to recover potentially lost intermediate data are re-executed.

  • No batching/scheduling overhead
  • Results are continuously updated

  • Dynamic resource allocation: resources can change over time between micro-batches

Mobile code paradigms represent architectural models in distributed systems based on the ability to transfer code, data, or execution state1 between nodes at runtime. The distinction between paradigms depends on the original and final location of three key elements: know-how (code), computational resources (CPU, memory, execution state), and data.

The Three Main Paradigms

Remote Evaluation In the Remote Evaluation paradigm, site A sends a request to site B and attaches its own code. Site B then processes the data using the received code and sends the results back to site A. The key characteristics are:

  • Code travels from client to server
  • Data resides on the server
  • Computational resources are on the server
  • Results are sent back to the client

Code on Demand In this paradigm, site A possesses the data and requests code from site B. Once received, A executes the code locally. A concrete example is JavaScript executed in web browsers: the browser (site A) downloads JavaScript code from the server (site B) and executes it locally on data available in the browser. The key characteristics are:

  • Code travels from server to client
  • Data resides on the client
  • Computational resources are on the client
  • It is an optional constraint of the REST architecture

Mobile Agent In the Mobile Agent paradigm, agent A migrates to site B bringing both code and computational resources (including execution state). Once arrived, the agent continues executing the code at the new site. The key characteristics are:

  • Both code and state migrate
  • The agent is autonomous and can move between multiple sites
  • Requires strong mobility in most cases

Types of Mobility

Mobile code technology is divided into two categories:

Strong mobility: The ability of a system to allow migration of both code and execution state to a different computational environment. Very few systems provide it, usually research-based.

Weak mobility: The ability of a system to allow only code movement across different computational environments. Supported by mainstream systems such as Java, .NET, and the Web.

Evolution of P2P Systems

The evolution of P2P file-sharing systems represents a continuous attempt to address fundamental challenges1: scalability, decentralization, robustness, and fairness.

Napster (1999): Centralized Search Napster was the first successful P2P file-sharing application, providing free music over the Internet. It pioneered the idea of sharing storage and bandwidth of individual home users.

Architecture: Napster used a centralized directory server for search, while file transfer occurred directly between peers2. The operations were:

  • Join: Clients contact a central server
  • Publish: Submit list of files to central server
  • Search: Query the server for files (O(1) search scope)
  • Fetch: Get files directly from peers

Problems: Single point of failure and control, server maintains O(N) state, server performs all search processing.

Gnutella (2000): Query Flooding Gnutella addressed Napster's centralization problem by eliminating the central server entirely.

Architecture: Fully decentralized using query flooding. Each query is forwarded to all neighbors, with propagation limited by a HopsToLive (HTL) field.

Operations:

  • Join: Contact a few known nodes via PING/PONG discovery protocol
  • Publish: No explicit publication needed
  • Search: Flood queries to neighbors (O(N) search scope, O(2D) search time where D is HTL). High expressivity (i.e. exact/fuzzy text matching)
  • Fetch: Direct download from peer

Problems: Massive network flooding—if average neighbors is C and HTL is D, each search generates CD messages. Network instability due to frequent node departures.

Kazaa (2001): Hierarchical Topology Kazaa introduced a hierarchical structure with superpeers to reduce flooding overhead.

Architecture: Distinction between normal nodes and supernodes (high-capacity, well-connected nodes). Query flooding occurs only among supernodes.

Operations:

  • Join: Contact a supernode; may become a supernode based on resources
  • Publish: Send file list to supernode
  • Search: Query supernode, which floods among other supernodes
  • Fetch: Direct download, can fetch from multiple peers simultaneously

Improvements: Considers node heterogeneity (bandwidth, computational resources, availability) and network locality. Problems: Still no guarantees on search scope or time.

BitTorrent (2001): Collaborative Sharing BitTorrent addressed the free-rider problem where users download without contributing.

Architecture: Uses a centralized tracker for peer discovery, but introduces incentive mechanisms through choking/unchoking.

Operations:

  • Join: Contact tracker server to get peer list
  • Publish: Run a tracker server
  • Search: Out-of-band (e.g., Google for .torrent files)
  • Fetch: Download chunks from peers using rarest-first strategy; upload to others

Key innovation: Tit-for-tat strategy—peers unchoke those providing the best download rates, with periodic optimistic unchoking to discover better peers. Problems: Central tracker dependency.

The Chord Algorithm

Chord is a Distributed Hash Table (DHT) protocol that provides a structured approach to resource location in P2P systems. It organizes nodes and data items in a logical ring structure using consistent hashing.

Basic Structure Nodes and keys are organized in a logical ring with identifiers in the range [0, 2m − 1], where m is the number of bits used for identifiers. Each node is assigned a unique m-bit identifier, typically computed as the hash of the node's IP address. Similarly, each data item is assigned a unique m-bit key, usually the hash of the item's name or content.

The fundamental rule in Chord is that an item with key1 k is stored at the node with the smallest4 identifier ≥ k, called the successor of k, denoted successor(k).

Finger Tables To enable efficient routing, each node maintains a finger table with m entries. Entry i in the finger table of node n points to the first node whose identifier is at least n + 2i−1 (for i = 1, 2, . . . , m).

More formally, the i-th finger of node n is:

finger[i] = successor(n + 2i−1)

This exponential spacing means the i-th finger points 1/2m−i of the way around the ring, allowing nodes to "skip" over large portions of the identifier space.

Routing Algorithm When a node n receives a query for key k:

  • If k falls between n and n's successor, n's successor is responsible for k
  • Otherwise, n forwards the query to the node in its finger table whose ID is closest to (but does not exceed) k

Since each hop roughly halves the remaining distance in the identifier space, routing takes O(log N) hops.

Joining Protocol When a new node n joins the network:

  • Initialize state: Node n contacts a bootstrap node n' already in the system
  • Build finger table: For each finger i, n uses n' to find successor(n + 2i−1), requiring O(log N) lookups per finger, for a total of O(m log N) = O(log2 N) messages
  • Update existing nodes: Nodes whose finger tables should point to n must be updated. Node n becomes the i-th finger of node p if and only if:
    • p precedes n by at least 2i−1
    • The i-th finger of p currently succeeds n

Finding and updating all affected nodes also costs O(log2 N)

Stabilization and Fault Tolerance To maintain correctness despite concurrent joins and departures, Chord runs periodic stabilization procedures. Each node periodically:

  • Verifies its successor and predecessor pointers
  • Refreshes its finger table entries by looking up successor(n + 2i−1)

For increased robustness, each node maintains a successor list of size R, containing its first R successors. If the immediate successor fails, the node can contact the next one in the list. This provides resilience even if R − 1 successive nodes fail simultaneously.

Complexity Summary

  • Routing table size: O(log N) entries (one per finger)
  • Routing time: O(log N) hops with high probability
  • Joining time: O(log2 N) messages with high probability

Why Chord is Better than Gnutella/Kazaa

Chord provides several advantages over earlier P2P systems:

  • Guaranteed lookup: Unlike Gnutella (which floods queries with limited TTL) and Kazaa (which uses hierarchical flooding), Chord guarantees that a lookup will succeed if the key exists, in O(log N) hops
  • Scalability: Gnutella generates O(CD) messages per query (where C is average neighbors and D is hops-to-live), which grows exponentially. Chord requires only O(log N) messages
  • Low per-node state: Each node stores only O(log N) entries, whereas maintaining complete knowledge would require O(N) state
  • Predictable performance: Search time and message complexity are deterministic, unlike flooding-based approaches where performance depends on network topology and TTL settings
  • Decentralization: Unlike Napster's centralized directory, Chord has no single point of failure

The main trade-off is that Chord only supports exact-match lookups, while flooding-based systems can support more expressive searches (fuzzy matching, keyword search, range queries).

Freenet vs. Chord

Freenet and Chord represent two different philosophies for P2P resource location2.

Freenet Characteristics Freenet is designed to ensure true freedom over the internet. Each node maintains routing information about its neighbors, associating keys with nodes that previously handled requests for similar keys. When searching for a key:

  • The query is forwarded to the neighbor whose routing table suggests it is "closest" to the target key
  • If a dead-end is reached, backtracking occurs
  • Data is cached at intermediate nodes along the retrieval path

Key features:

  • Anonymity: Designed for censorship resistance; intermediate nodes cannot determine the origin or destination of requests
  • Adaptive routing: Popular content migrates toward areas of high demand through caching
  • Small-world properties: Network evolves such that most queries traverse only a few hops
  • No guarantees: Lookup may fail if the content doesn't exist or if the TTL expires

Why Query Flooding Enables Content-Based Search

The fundamental difference between query flooding and DHT-based systems lies in how they locate resources.

Query Flooding (Gnutella, Kazaa) In query flooding systems:

  • Queries are broadcast to all (or many) nodes in the network
  • Each node examines its local data to determine if it matches the query
  • Queries can be arbitrarily complex: text matching, regular expressions, fuzzy matching, metadata search (artist, genre, date), range queries

Because every node evaluates the query against its local data, any search expressivity supported by local evaluation3 is supported by the system. The query doesn't need to specify an exact identifier—it can describe desired properties, and all matching items will be returned.

Structured DHT Systems (Chord) In Chord and similar DHTs:

  • Data is stored at deterministic locations based on a hash of the data's key
  • To find an item, you must know (or compute) its exact key
  • The lookup process routes directly to the node responsible for that key

This design provides efficient, guaranteed lookup for exact-match queries only. However, it makes content-based search difficult because:

  • Hash functions destroy semantic relationships: Similar keys (e.g., "Distributed Systems" and "Distributed Computing") hash to completely unrelated identifiers. There's no way to find "nearby" content
  • No local evaluation opportunity: Queries are routed to a single node based on the key. Intermediate nodes don't examine the query or their local data
  • Range queries are inefficient: To find all items in a range, you would need to query multiple nodes or flood the network, defeating the purpose of the DHT structure

When to Use Chord: Scenario Analysis

Scenario 1: Tens of Thousands of Hosts, Dynamic Join/Leave Use Chord (or another DHT).

Reasoning: Chord is specifically designed for dynamic networks5 where nodes frequently join and leave, providing mechanisms like stabilization and successor lists to handle churn. The O(log N) complexity for routing and table size becomes efficient when N is large—with 100,000 nodes, only 17 hops and 17 routing entries are needed. Despite being somewhat fragile to rapid changes, Chord's decentralized structure and guaranteed lookup make it suitable for large-scale dynamic environments.

Scenario 2: Less than 100 Hosts, No Join/Leave Use a simple full replication approach (distributed database model).

Reasoning: With fewer than 100 nodes and static membership, Chord's complexity is unnecessary. A simpler solution is to maintain a complete routing table at each node6—a table with 100 entries listing all nodes and their data. This fits easily in memory and provides O(1) lookup. Since membership is static, the routing table never changes, eliminating the need for stabilization protocols. This approach is essentially how small distributed databases operate: everybody knows about everybody else7.

BitTorrent Protocol Overview

BitTorrent breaks files into small chunks (typically 256KB) and enables peers to download from each other. Each peer downloads missing fragments from multiple peers and uploads fragments it possesses to others. Peers use a rarest-first strategy: they prioritize downloading fragments held by the fewest peers to maintain high availability across the swarm.

BitTorrent operates through the following components and operations:

  • Torrent file: A meta-data file containing file names, sizes, checksums of all blocks (files are split into fixed-size chunks, typically 256KB), tracker address, and peer addresses
  • Tracker: A centralized server that maintains a list of peers participating in the swarm (all peers sharing a particular file)
  • Seed: A peer that has the complete file and continues to offer it for upload
  • Leech: A peer with an incomplete download

Operations:

  • Join: Contact the centralized tracker server to obtain a list of peers in the swarm
  • Publish: Run a tracker server
  • Search: Out-of-band (e.g., use Google to find a .torrent file)
  • Fetch: Download chunks from peers; upload chunks you possess to others

The Choking Mechanism

The choking mechanism implements a tit-for-tat incentive system to combat free-riding1. Each peer:

  • Un-chokes (uploads to) the top 4 peers based on download rate from them, re-evaluated every 10 seconds
  • Maintains one optimistic unchoke—a random peer uploaded to regardless of rate, rotating every 30 seconds

The principle: I upload to whoever downloads from me fastest.

Optimistic unchoking serves two purposes: bootstrap new peers that have no data to trade, and discover better connections.

Problem Solved: Free-Riding

In earlier P2P systems (Napster, Gnutella), roughly 70% of users shared no files, with nearly 50% of responses from just the top 1% of contributors. This free-rider problem reduced system performance and was unfair to contributors.

Previous attempts failed:

  • Napster: Voluntary enforcement; easily ignored
  • KaZaA/eMule: Credit-based systems; easily circumvented with modified clients reporting false credits

BitTorrent's advantage: real-time enforcement based on actual measured rates, built into the protocol itself. Free-riders experience immediate poor download performance, creating strong incentives to contribute. The mechanism is harder to circumvent because it measures objective facts (bytes transferred), not subjective claims.

Best case: small quorums give low latency; worst case: large quorums approaching N give strong consistency but high latency.

Describe and compare single leader, multi leader and leaderless protocols
Q 5.1 · complete answer 2 of 4
Complete answer — in plain words

These three architectures differ in who may accept writes. In a Single Leader model one replica is the leader: all writes go only to it, it applies them locally first and then propagates to the followers, while reads can be served by any replica. This gives a natural global order but the leader is a bottleneck. In a Multi-Leader model several replicas act as leaders and accept writes concurrently; since no single entity orders all writes, the same data modified on different leaders can cause write-write conflicts, and how they are resolved depends on the consistency model. In a Leaderless model the client directly contacts several replicas and uses quorums: a write succeeds only when a majority of replicas agree, and a read needs a quorum of replicas to agree on a value. So the trade-off is ordering simplicity (single leader) versus concurrency and conflict handling (multi/leaderless).

When is a consistency model for replication highly available? Which is highly available: FIFO, causal, sequential, linearizable? Under which assumptions?
Q 5.1 · complete answer 3 of 4
Complete answer — in plain words

A protocol is Highly Available (HA) if it does not require synchronous, blocking communication, so a client can still get a response from a working replica even when other replicas are unreachable or the network is partitioned. Out of the four models: Causal and FIFO (PRAM) are highly available; Sequential and Linearizable are NOT. Causal is in fact the strongest HA model because it avoids synchronous coordination, typically using vector clocks to track dependencies without waiting on network delays. FIFO is also HA and even simpler, needing only sequence numbers to preserve each sender's write order. Linearizability requires costly synchronous protocols to enforce a global real-time order, and Sequential also needs synchronous updates so everyone sees the same global order; both therefore block during partitions. The assumption behind this is that availability is judged under network partitions, where blocking models cannot respond.

Describe the client-centric consistency models.
Q 5.1 · complete answer 4 of 4
Complete answer — in plain words

Client-centric consistency models were created to fix inconsistencies seen by a single client interacting with a replicated store, especially when mobility or variable connectivity makes the client connect to different replicas over time. The core problem: if a client switches replica and its earlier updates haven't propagated to the new one, it would seem as if its own changes never happened. They guarantee four properties: Monotonic Reads (after reading a value, later reads return that value or a newer one, never older); Monotonic Writes (a write completes before the same client's next write on that item, so its writes apply everywhere in order); Read Your Writes (a write is always seen by the same client's later read, on any replica); and Writes Follow Reads (a write after a read is applied to a value at least as recent as the one read). They are enforced by tracking each client's Read-Set and Write-Set (often as vector clocks).

Does eventual consistency imply sequential consistency?
Q 5.3 · complete answer 1 of 4
Complete answer — in plain words

No. Eventual consistency (EC) does not imply sequential consistency; EC is strictly weaker. EC only promises that if writes stop, all replicas eventually converge to the same value. It says nothing about the order in which writes are observed, either across different processes or even by a single process over time. Sequential consistency, by contrast, demands that all operations appear to execute in some single total order that is consistent with each process's program order, and that every process sees that same interleaving. EC happily allows temporary inconsistencies and observation orders that would violate sequential consistency, so it cannot guarantee it. In short, convergence-in-the-limit is a much weaker promise than a globally agreed total order.

Does eventual consistency imply causal consistency?
Q 5.3 · complete answer 2 of 4
Complete answer — in plain words

No. Eventual consistency (EC) does not imply causal consistency. EC guarantees only that replicas eventually converge to the same value once writes stop; it does not respect the order of causally related operations. Because of replication lag, an EC system can let a process observe an effect before its cause — for example seeing a write before the write that influenced it — which directly violates causal consistency. Causal consistency requires that the potential causal order (Lamport's happens-before relationship) is respected on every replica: causally related writes must be seen in the same order everywhere, while only truly concurrent writes may be reordered. Since EC enforces no such ordering, it is strictly weaker and cannot guarantee causal consistency.

Does eventual consistency imply FIFO consistency?
Q 5.3 · complete answer 3 of 4
Complete answer — in plain words

No. Eventual consistency (EC) does not imply FIFO (PRAM) consistency. EC only guarantees that replicas will eventually converge to the same value if writes stop; it does not guarantee that writes from a single process are observed in the order that process issued them. In an EC system a client reading from different replicas might see newer data and then later see older data (violating Monotonic Reads), which is incompatible with FIFO consistency. FIFO consistency specifically requires that all writes coming from the same source are seen by every observer in their original issuance order, even though writes from different sources may be interleaved arbitrarily. Because EC imposes no per-sender ordering, it is weaker and cannot guarantee FIFO.

Consider a messaging system implemented on top of an eventually consistent datastore. How could you make the system FIFO consistent? How could you you make the system causally consistent?
Q 5.3 · complete answer 4 of 4
Complete answer — in plain words

An EC datastore only guarantees eventual convergence, so any ordering must be added at a higher layer (application or middleware). To make it FIFO consistent (messages from the same sender delivered in send order, different senders interleaved freely): attach an increasing sequence number to each message; on the receiver side, messages arriving out of order are held in a hold-back queue until the missing predecessors arrive. To make it causally consistent (causally related messages delivered in happens-before order, concurrent ones in any order): use vector clocks. Each process Pi keeps a vector Vi; on send it increments Vi[i] and attaches the whole vector as timestamp tm. A receiver Pj delays delivering m from Pi until (1) tm[i] = Vj[i] + 1 (it is the next message from Pi) and (2) tm[k] ≤ Vj[k] for all k ≠ i (it has all messages Pi knew about). Messages failing these are buffered until their dependencies are met.

MapReduce computes average product ratings from tuples ⟨user, product, rate⟩ in two user-written functions plus an automatic shuffle. Because an average needs both the sum and the count of ratings per product, the map emits both. Map: for each record, take the product as the key and emit a pair carrying the rating and a count of 1, i.e. emit(product, [rate, 1]). Emitting [rate, 1] (rather than just the rate) is what lets the reducer recover both the sum and the count needed for the average. Map and reduce tasks run in parallel, giving automatic parallelization and fault tolerance.

Consider the dataflow model for big data processing.
Q 6.2 · complete answer 1 of 2
Complete answer — in plain words

The dataflow programming model generalizes MapReduce. Where MapReduce allows only two stages (map then reduce), dataflow expresses a computation as an arbitrary directed acyclic graph (DAG) of transformation operators — each node a transformation, each edge a data flow — so complex pipelines need not be split into many chained jobs. Its key characteristics: a unified abstraction for both batch and stream processing on the same engine; immutable transformations (operators produce new outputs instead of mutating shared state), which eases reasoning and parallelization; automatic parallelization, where the runtime decomposes logical operators into parallel physical tasks; a declarative interface using map, filter, reduce, join, aggregate; built-in fault tolerance via lineage tracking or distributed snapshots; and data-locality optimization, scheduling tasks near their input to cut network traffic. Concrete engines include Apache Spark (stage scheduling) and Apache Flink (pipelining).

Provide a brief overview of the pipelining architecture for implementing a big data processing system, focusing on the following aspects:
Q 6.2 · complete answer 2 of 2
Complete answer — in plain words

The pipelining architecture (e.g. Apache Flink) executes a big-data job as a continuous pipeline. Execution model: at job submission all tasks for all stages are instantiated and deployed; they communicate over persistent TCP channels forming a topology matching the dataflow graph. A task starts as soon as it has data from upstream — it does not wait for upstream stages to finish — and data flows through without being materialized to storage between stages, so operators from different stages run simultaneously. Advantages: low latency (no scheduling or intermediate-I/O overhead, ideal for subsecond stream processing), efficient resource use (all stages active at once), and natural streaming support. Disadvantages: no runtime elasticity (resources fixed at submission), limited dynamic load balancing, and coarse-grained fault tolerance. Recovery uses distributed snapshots (Chandy-Lamport style): source operators inject markers; a task that has received markers on all input channels saves its state and forwards the marker, building a global snapshot. On failure the whole job restarts from the last snapshot and replays data processed since.

Describe the different mobile code paradigms.
Q 7.1 · complete answer 1 of 2
Complete answer — in plain words

Mobile code paradigms are architectural models in distributed systems based on transferring code, data, or execution state between nodes at runtime. Which paradigm you have depends on the original and final location of three elements: know-how (code), computational resources (CPU, memory, state) and data. There are three main paradigms:

  • Remote Evaluation: site A sends a request to site B attaching its own code; B holds the data and resources, runs the code, and returns results to A.
  • Code on Demand: site A already has the data and resources but requests the code from B, then executes it locally (e.g. JavaScript downloaded by a browser).
  • Mobile Agent: an autonomous agent migrates to site B carrying both code and execution state, and keeps running there; it usually needs strong mobility.

Mobility is either strong (code + execution state move) or weak (only code moves, as in Java, .NET, the Web).

Describe the REST architectural style
Q 7.2 · complete answer 1 of 1
Complete answer — in plain words

REST (REpresentational State Transfer), introduced by Roy Fielding, is both a descriptive framework for the Web and a set of architectural principles for how Web standards should be used. Its goals are scalability of interactions, generality of interfaces, independent deployment of components, and useful intermediaries. It is defined by six constraints:

  • Client-Server with separation of concerns;
  • Stateless — every request carries all info needed, state is transferred from clients to servers;
  • Cacheable — responses are labeled (non-)cacheable, which is key to scalability;
  • Layered System — a component sees only the layer it talks to;
  • Code-on-Demand (optional) — servers may ship executable code to clients;
  • Uniform Interface.

The uniform interface adds four sub-constraints: identification of resources (each has a URI), manipulation through representations (XML/JSON), self-descriptive messages, and HATEOAS (clients move between states via hypermedia links).

Briefly illustrate the chord algorithm, explain the routing table size and the routing and joining time complexity. Explain why this algorithm is better than those historically used in p2p systems (e.g., Gnutella, Kazaa).
Q 7.5 · complete answer 1 of 7
Complete answer — in plain words

Chord is a DHT that places nodes and keys on a logical ring of identifiers in [0, 2^m − 1] using consistent hashing; a key k is stored at its successor, the smallest node ID >= k. Each node keeps a finger table of m entries where finger[i] = successor(n + 2^(i−1)), giving exponentially spaced shortcuts. Routing forwards a query to the finger closest to but not past the key, halving the distance each hop.

Complexity: routing-table size O(log N), routing O(log N) hops, joining O(log^2 N) messages (building fingers and updating affected nodes). It beats Gnutella and Kazaa because it guarantees a lookup succeeds in O(log N) hops, needs only O(log N) state per node instead of O(N), and gives predictable, decentralized performance — whereas flooding costs O(C^D) messages. The trade-off: Chord supports only exact-match lookups.

Q 7.5 · complete answer 2 of 7
Complete answer — in plain words

Freenet and Chord embody two philosophies. Freenet is unstructured and self-organizing: each node keeps routing hints associating keys with neighbors that handled similar keys, forwards a query to the "closest" neighbor, backtracks on dead-ends, and caches data along the retrieval path. Chord is structured: keys are placed deterministically at their successor, routing uses finger tables, and lookups are guaranteed to succeed in O(log N) hops, but there is no anonymity (nodes know query origin/destination) and resilience requires stabilization to maintain the ring. In short, Freenet prioritizes anonymity and censorship resistance at the cost of guarantees, while Chord prioritizes efficient, predictable lookup with no privacy.

With query flooding (Gnutella, Kazaa) a query is broadcast to many nodes and each node evaluates it against its own local data.
